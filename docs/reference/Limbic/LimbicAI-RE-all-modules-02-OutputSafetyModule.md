**OutputSafetyModule**

它是**输出侧安全闸门**。职责是：

- 在 `LanguageModel` 已经生成 `system response` 之后
- 在真正发给用户之前
- 再检查一遍这条回复是否安全、合适、可放行

和 `SubjectSafetyModule` 的区别：

- `SubjectSafetyModule` 看的是**用户输入**
- `OutputSafetyModule` 看的是**系统输出**

---

**1. 输入信息 / 变量**

核心输入：

1. `system_response_text`
   - 由 `LanguageModel` 生成的文本回复
   - 专利原文最明确的输入对象就是这个

2. 可选输入：安全规则资源
   - `unsafe_words`
   - `unsafe_phrases`
   - `regex_patterns`
   - 用于检测输出中不该出现的内容

3. 可选输入：输出安全分类模型
   - 一个 trained model
   - 输入系统回复文本
   - 输出风险标签，例如：
     - `risk`
     - `no risk`

4. 可选输入：目标主题 / 安全约束
   - 专利里提到可用 prompted LLM 判断内容是否 “on topic”
   - 所以也可理解为存在：
     - `desired_topic`
     - `allowed_scope`

5. 可选输入：人工升级配置
   - therapist / human operator 联系方式
   - rules-based flow 配置
   - fallback prompt editing logic

**可以抽象成变量**

- `llm_output_text`
- `output_safety_keywords`
- `output_safety_phrases`
- `output_safety_regex`
- `output_risk_classifier`
- `desired_topic`
- `handoff_config`

---

**2. 中间运行机制**

专利里给的也是两大类主机制，加一个可选 LLM 方案。

**机制 A：规则匹配 / 关键词扫描**

运行过程：

1. 拿到 `LanguageModel` 输出的 `system_response_text`
2. 在真正发给用户前进行扫描
3. 搜索预定义的词或短语
4. 可通过一个或多个正则表达式匹配
5. 如果命中：
   - 判定输出有风险
   - 阻止该回复发送给用户

这是最直接的输出过滤器。

---

**机制 B：训练模型进行输出安全分类**

运行过程：

1. 把 `system_response_text` 输入 trained model
2. 模型输出标签：
   - `risk`
   - `no risk`
3. 如果是 `risk`
   - 阻止回复输出给用户

专利里给的模型示例：

- `feed forward neural network`
- 输入：`system response text`
- 训练数据：由 trained raters 标注的 utterances
- 损失：`cross-entropy loss`

---

**机制 C：prompted LLM 作为输出审查器**

专利里补充提到：

1. 用 prompted LLM 判断输出是否“on topic”
2. 再结合标注数据和内部测试做验证

这个更像一种可替代实现，不是主路径。

---

**3. 风险命中后的处理机制**

一旦 `OutputSafetyModule` 认为当前回复不安全，专利里提到的后续动作有几种：

1. **阻止当前回复发送**
   - 最核心动作

2. **编辑 system prompt 后重新送回 LanguageModel**
   - 即：
     - 修改 prompt
     - 再次生成一个更安全的回复

3. **切换到 rules-based dialogue flow**
   - 不再依赖当前 LLM 输出
   - 走预定义安全流程

4. **交给 human operator**
   - 人工接管

5. **通知 therapist**
   - 通过 email / text / phone call / alert 等方式

所以它不是只有“放行/拦截”，还带有**故障转移和升级路径**。

---

**4. 中间决策变量**

你整理时可以把它抽象成这些中间状态：

- `output_risk_detected`
- `matched_unsafe_pattern`
- `output_risk_score`
- `output_risk_label`
- `allow_output`
- `requires_regeneration`
- `route_to_rules_flow`
- `route_to_human_operator`
- `notify_therapist`

---

**5. 输出信息 / 变量**

**正常情况输出**

如果没检测到风险：

- `output_safety_status = pass`
- `allow_output = true`
- 系统回复可以发给用户

可抽象为：

- `approved_response_text`
- `send_to_user = true`

---

**风险情况输出**

如果检测到风险：

1. 控制输出
   - `output_safety_status = risk`
   - `allow_output = false`
   - 当前 `system_response_text` 不发送

2. 重新生成路径
   - `requires_prompt_edit = true`
   - `regenerate_response = true`

3. 路由路径
   - `route_to_rules_flow = true`
   - 或 `route_to_human_operator = true`

4. 告警路径
   - `notify_therapist = true`

所以输出既可能是一个布尔决策，也可能是一个更完整的控制包。

**可整理成变量**

- `output_safety_status`
- `approved_response_text`
- `block_output`
- `edited_prompt_needed`
- `regeneration_requested`
- `fallback_route`
- `escalation_action`

---

**6. 一句话版运行链路**

`LanguageModel system_response`
-> `OutputSafetyModule`
-> 规则/模型审查
-> 若安全：发给用户
-> 若不安全：阻断，并选择重新生成 / 规则流 / 人工升级 / 通知治疗师

---

**7. 建议你文档里直接用的结构化模板**

```text
Module: OutputSafetyModule

Inputs:
- system_response_text generated by the language model
- pre-defined unsafe words / phrases
- regex patterns
- optional trained output-risk classifier
- optional desired-topic / safety constraints
- optional handoff / escalation configuration

Mechanism:
- evaluate language-model output before sending it to the user
- detect unsafe output via:
  - keyword / phrase matching
  - regex matching
  - optional trained classifier
  - optional prompted LLM judge
- if unsafe:
  - block current response
  - optionally edit prompt and regenerate
  - optionally route to rules-based flow
  - optionally hand off to human operator
  - optionally notify therapist

Outputs:
- output safety decision: pass / risk
- allow_output flag
- optional blocked_response marker
- optional regenerate / edit-prompt instruction
- optional fallback route
- optional escalation / therapist notification action
```
