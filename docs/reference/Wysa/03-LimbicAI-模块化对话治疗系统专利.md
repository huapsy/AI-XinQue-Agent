# LimbicAI - 模块化对话治疗系统

> 来源: US 2026/0024525 A1, Limbic Ltd, 2026-01-22
> 类型: 美国专利申请

## 核心思想

将心理治疗对话系统设计为**两部分模块化架构**：
1. **第一模块（理解+推荐）** - 多个小型专用模型，负责理解用户状态并推荐干预
2. **第二模块（语言模型）** - 冻结的通用大语言模型，负责生成共情对话

用小模型群赋予大语言模型"治疗师的知识"，而非依赖微调。

## 系统架构

### 整体流程

```
用户输入 → [第一模块: 理解+推荐] → 生成 System Prompt → [第二模块: LLM] → 系统回复
              ↓                                                    ↑
         用户画像信息 ─────────────────────────────────────────────────┘
```

### 第一模块：主体理解模块 (Subject Understanding Module)

由多个小型专用模型组成，每个模型专注于临床过程的一个方面：

| 子模型 | 功能 | 参数规模 |
|--------|------|---------|
| **认知扭曲模型** | 识别认知扭曲类型 | < 1M 参数 |
| **想法检测模型** | 判断信息是否对应一个"想法" | 小型 |
| **扭曲思维模型** | 判断想法是否为扭曲思维 | 小型 |
| **情感模型** | 判断正面/负面情感 | 小型 |
| **活动检测模型** | 判断信息是否对应一个活动 | 小型 |
| **行为理解模型** | 将行为分类为适应性/不适应性 | 小型 |

### 认知扭曲类型

系统可识别的认知扭曲：
- **灾难化思维 (Catastrophizing)**
- **非黑即白思维 (Dichotomous thinking)**
- **消极过滤 (Negative filtering)**
- **算命思维 (Fortune telling)**
- **读心术 (Mind reading)**
- **个人化 (Personalising)**

### 第二模块：大语言模型

- 冻结的、通用的、预训练 GPT 式 Transformer
- 不需要微调（微调需要 40,000+ 小时的治疗数据，计算成本极高）
- 通过 System Prompt 接收第一模块的分析结果
- 负责生成共情、个性化的对话回复

### System Prompt 构成

System Prompt 由以下信息组合生成：

```
[临床知识] 抑郁症患者通常表现出5种扭曲思维模式
[干预建议] 认知重构是针对扭曲思维的适当练习
[重构示例] 成功重构的扭曲想法示例如下：...
[用户画像] 当前用户的认知扭曲类型、情感状态、行为模式
[对话目标] 本次对话的治疗目标
```

## 关键设计优势

### 1. 模块化 vs 单一大模型

| 维度 | 单一大模型 | 模块化系统 |
|------|----------|----------|
| 微调成本 | 需要海量治疗数据（40k+ 小时） | 各子模型可独立小数据训练 |
| 可解释性 | 黑盒，无法追溯决策依据 | 每个子模型输出可审计 |
| 可扩展性 | 改一处影响全部 | 新增病症只需添加新子模型 |
| 目标导向 | 无内在治疗目标 | 第一模块提供目标和方向 |
| 长期记忆 | 受上下文窗口限制 | 子模型持续积累用户洞察 |
| 事实知识 | 可能幻觉 | 由子模型注入临床事实 |

### 2. 可解释性

- 子模型的输出可直接生成**会话报告**
- 临床医生可查看每轮对话触发了哪些临床概念
- 可审计和修正每个子模型的临床逻辑

### 3. 治疗联盟

在两个层面建立：
- **长期层面** - 用户感到被理解，认为任务和练习有用
- **短期层面** - 对话具有共情性和参与感（镜像语气、积极倾听）

## 安全机制

### 输入安全模块 (Subject Safety Module)
- 在 LLM 回复前评估用户输入
- 检测危机信号（自杀、自伤等）
- 可阻止系统生成不当回复

### 输出安全模块 (Output Safety Module)
- 在 LLM 回复输出前评估
- 过滤不安全或不适当的内容
- 可阻止不当内容发送给用户

## 交互模块 (Interaction Module)

### 结构化信息收集

交互模块生成问题并按**主题标签**组织：
- **想法 (Thoughts)** - "当你感到这样时，脑海中出现了什么想法？"
- **情境 (Situations)** - "是什么情况让你有这种感觉？"
- **感受 (Feelings)** - "这让你感觉如何？"

### 临床状态检测流程

```
用户输入 → 第一个 System Prompt（判断是否包含临床信息）
                ↓
         包含临床信息？ ──是──→ 标记 Flag 1 → 生成第二个 System Prompt → 提问收集更多信息
                ↓
               否 → 普通对话回复
```

```
用户输入 → 判断是否请求了特定干预
                ↓
         请求干预？ ──是──→ 标记 Flag 2 → 直接执行干预
                ↓
               否 → 继续收集信息
```

## 推荐模块 (Recommender Module)

### 推荐依据
- 存储的诊断信息
- 用户人口统计数据
- 历史干预记录
- 用户对以往干预的反馈

### 优化目标（三个指标取其最优）
1. **治疗联盟** - 用户与系统的信任和连接
2. **患者疗效** - 实际的临床改善
3. **理解模块洞察** - 子模型派生的理解深度

### 干预序列
系统回复以**干预序列**形式交付，而非单次干预。

## 对项目的启发

### 架构参考

1. **两层架构** - 可直接参考：用小型分类模型做理解层 + GPT-5.x 做对话层
2. **System Prompt 动态生成** - 将用户状态分析结果注入 System Prompt，引导 LLM 行为
3. **输入输出双重安全检查** - 在 LLM 前后各设一道安全关卡

### 临床模型借鉴

1. **认知扭曲识别** - 6 种扭曲类型可作为意图分类的参考
2. **想法-情感-行为三维度** - 结构化收集用户信息
3. **临床知识注入 Prompt** - 将 CBT 等治疗协议作为 Prompt 的一部分

### 实际可行的简化方案

鉴于项目使用 Azure OpenAI API，可以：
- 用 GPT-5.x 本身替代多个小型子模型（通过不同的 System Prompt 实现多角色）
- 或用一次 Function Calling 完成多维度分析
- 保留"理解层 → Prompt 生成 → 对话层"的核心架构思路
- 安全检查可通过 Azure Content Safety API 辅助实现
