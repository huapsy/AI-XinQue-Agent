# Plan-v2.1 Production Gap Plan

## 背景

截至 `Sprint 35`，心雀已经完成 `plan-v2.1` 的 MVP 主线能力：单一核心 Agent、Responses-first 运行时、核心 Tool 集、结构化状态、分层记忆、基础可观测性、Prompt contract 与评估闭环均已落地。

当前剩余工作，不再是“功能缺失导致主链路不可用”，而是“从已可运行的产品，推进到更稳健的生产与企业化形态”。这些工作跨多个 sprint，彼此存在依赖，不适合继续零散塞进单个 `spec`。

因此，需要先形成一个新的跨 sprint 实施计划，再按 Harness 方式拆成新的 sprint specs。

## 目标

基于 [`product-plan-v2.1.md`](/E:/AI_XinQue_Agent/docs/design/product-plan-v2.1.md) 与当前实现状态，整理出下一阶段的开发主线，覆盖以下剩余高优先级差距：

- Prompt 体系继续升级，收口为更强的契约化、分层化与 evaluator 可消费规则
- 更细粒度的 `phase/state timeline` 聚合与运营分析
- 更强的主题连续性重排与跨 skill 冷却策略
- 外部向量索引 / 检索抽象升级
- 生产级 OTel / Dashboard 接入
- 企业级密钥轮换、多租户与权限边界

## 规划原则

### 1. 先补“消费层”和“治理层”，再补“基础设施替换”

当前很多底层能力已经存在最小版，但运营读取、分析、治理仍偏薄。优先补消费层与治理层，能最快提升产品可运营性，也能为后续基础设施升级提供观测与验证基线。

### 2. 不在同一 sprint 同时做两类高风险重构

外部向量索引、OTel 平台、企业级密钥体系都带来明显集成风险。每个 sprint 只聚焦一类主变化，避免难以定位回归来源。

### 3. 继续沿用 Harness 交付模式

每个 sprint 仍然输出：

- `specs/sprint-XX-*.md`
- `contracts/sprint-XX-contract.md`
- `evaluations/sprint-XX-eval.md`

本计划只回答“接下来多个 sprint 怎么做”，不直接替代单个 sprint spec。

## 下一阶段建议切片

### Sprint 36：Prompt 体系升级与评估闭环深化

虽然 `Sprint 35` 已完成 Prompt 指南的首轮落地，但当前仍主要解决了“把指南写进 contract 和测试”。若按 `plan-v2.1` 的标准继续推进，下一阶段还应把 Prompt 升级为更强的运行时体系，包括更明确的契约分层、工具依赖、完成/验证闸门、长会话漂移控制，以及 evaluator 可直接消费的 prompt 审查信号。

### Sprint 37：状态时间线聚合与运营分析

补齐当前已经能读、但还不能很好消费的 `SessionState` / `phase_timeline` 数据，把“原始 trace 和历史状态”升级成可用于运营、调试、回访分析的聚合接口与摘要视图。

### Sprint 38：主题连续性重排与跨 Skill 冷却治理

围绕 `plan-v2.1` 中的长会话与多轮干预要求，提升心雀在“回访时抓主线”“避免重复推荐同类干预”“合理跟进近期作业/练习”上的一致性。

### Sprint 39：外部向量索引抽象与检索升级

把当前本地 embedding + 混合检索升级为“可切换的检索层”，优先补抽象、索引同步、回退策略与评估基线，不强行在一个 sprint 内绑定单一生产供应商。

### Sprint 40：生产级可观测平台接入

在已有 trace sink 与 OTel 兼容 exporter 的基础上，把心雀运行时接到完整 collector / dashboard / 查询面板，形成更像生产系统的观测链路。

### Sprint 41：企业级密钥轮换与租户权限基线

把“应用层加密”推进到“企业可交付的安全基线”，至少补齐密钥版本、轮换流程、租户边界、管理侧权限与审计准备。

## Sprint 依赖关系

### Sprint 36 → Sprint 37

更强的 timeline/state 聚合与长会话治理，应建立在更清晰的 Prompt 契约之上，尤其是 phase、完成闸门、工具依赖和 evaluator 审查信号。

### Sprint 37 → Sprint 38

主题连续性重排需要依赖更稳定的时间线、状态变化与近期干预消费接口，因此先做状态时间线聚合。

### Sprint 38 → Sprint 39

外部向量索引升级前，先明确“检索究竟要优化什么”。Sprint 38 会收口主题连续性和冷却治理，能减少把产品策略问题误判为向量索引问题。

### Sprint 39 → Sprint 40

检索层升级会带来新的延迟、失败模式和同步问题，先落地检索抽象，再建设生产级 dashboard，避免观测对象不稳定。

### Sprint 40 → Sprint 41

企业级安全与权限改造需要更完整的运行观测和操作审计基础，生产级可观测平台应先一步落地。

## 非目标

- 本轮计划不重写心雀核心对话范式，不回退单 Agent + Tool Use 架构
- 不在本计划内切换数据库主存储到 PostgreSQL
- 不把“更多 skill 内容扩充”作为主线优先级，除非某 sprint 为实现治理逻辑需要补最小样本
- 不把管理后台大改版作为下一阶段主目标

## 完成定义

如果本计划执行完成，心雀将从“主线能力完整的智能体产品”推进到“具备较强生产与企业交付准备度的智能体系统”，并且所有新增工作仍保持 Harness 风格：

- 先有跨 sprint 计划
- 再有单 sprint spec
- 再协商 contract
- 再实施与评估

## 推荐顺序

推荐采用以下顺序启动：

1. 先评审本计划，确认 `Sprint 36-41` 切法
2. 按本计划评审对应 specs
3. 确认要先启动的 sprint
4. 再由 Generator / Evaluator 协商该 sprint contract
