# Plan-v2.2 主Agent + 四阶段子Agent架构 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将心雀从“单一主 prompt + 自主阶段判断”演进为“主 agent 编排 + P1/P2/P3/P4 四阶段子 agent profile”架构，在不退回旧式多 Agent 管道的前提下，让阶段推进、Prompt 控制、Tool 权限和评估边界更清晰。

**Architecture:** 保留一个对用户说话的主 agent，不引入 5 个运行时独立 LLM 会话。主 agent 负责安全、状态、路由、上下文装配和 tool runtime；四个子 agent 以 phase profile / prompt block / flow markers 的方式存在，由主 agent 每轮选择激活。运行时仍使用 Responses API + tool loop，但 phase guidance 从单体 system prompt 拆成 base contract + active phase contract + global state markers。

**Tech Stack:** Python 3.13, FastAPI, Azure OpenAI Responses API, SQLAlchemy, unittest, React/Vite（前端仅必要联动）

---

## 背景与约束

- 当前权威设计仍是 `docs/design/product-plan-v2.1.md`；本计划是跨多个 sprint 的实施路线，不直接替代权威设计。
- 该架构调整属于长期运行时规则变更；实现落地后，需把稳定规则回写到 `docs/design/product-plan-v2.1.md` 或后续 `product-plan-v2.2.md`。
- 不把四阶段做成 4 个独立运行时聊天机器人，避免回退到 v1 多 Agent 管道。
- 不移除现有 tool runtime guardrail；新的 phase 子 agent 只能影响 prompt 组织与路由判断，不能替代安全层和 preflight。
- 与 Limbic 对齐的重点是“每阶段独立 prompt guidance + 少量 flow variables + 本地路由逻辑”，不是照搬其 52a-52d 状态机。

## 目标产物

- 新的 phase-agent prompt 组织结构
- 最小 flow state variables 定义
- 主 agent 路由器与 phase profile 选择逻辑
- phase-aware tool 权限与 guardrail
- trace / eval 可见的 phase timeline 与 phase routing evidence
- 覆盖新架构的后端单元测试
- 至少一个 sprint spec / contract / eval 证明该架构首轮落地

## 非目标

- 不引入真正的多模型并发运行时
- 不重写全部 skill 内容
- 不一次性实现 step-level workflow engine
- 不在本轮改造前端主界面交互结构

## 关键设计决定

### 1. 子 agent 不是独立会话

- 仍然只有一个对用户输出的主 agent
- 四个子 agent 是 phase profile，不是多轮轮换的独立人格

### 2. Prompt 组织从“单体字符串”变为“基座 + 单 phase block”

- Base contract 永远注入
- 当前 phase profile 只注入一个
- `active_skill`、alignment、risk 继续由 working contract / working context 动态注入

### 3. 路由以少量字段驱动，而不是大状态机

第一版只要求这些字段可用：

- `active_phase`
- `intent`
- `explore`
- `asking`
- `formulation_confirmed`
- `needs_more_exploration`
- `chosen_intervention`
- `intervention_complete`
- `active_skill`

### 4. Guardrail 优先级高于 phase prompt

优先级顺序固定为：

1. 输入安全
2. phase-aware tool guardrail
3. formulation / acceptance / active_skill guardrail
4. phase profile prompt guidance
5. 模型自由生成

## 风险与回退策略

- 风险 1：phase profile 过强，导致回复变僵
  - 回退：保留 base contract 优先，缩短 phase block，只把它用于工具和推进纪律

- 风险 2：phase-aware guardrail 过严，导致自然跳步能力下降
  - 回退：保留 direct intent -> P4、misalignment -> P1、active skill -> P4 三条高优先级例外

- 风险 3：active_phase 与 active_skill 双状态不一致
  - 回退：规定 `active_skill` 存在时，`active_phase` 强制落在 `p4_interventor`

- 风险 4：设计复杂度回退到 v1 多 Agent 管道
  - 回退：禁止真实多 LLM runtime handoff；只允许单 agent orchestrator + profiles

---

## 完成定义

该计划完成执行后，应满足：

- 心雀仍是单一主 agent 对外说话
- 四阶段各自有独立 phase profile，而不是混在一个 prompt 大段里
- phase routing 有最小状态字段支撑，而不是纯 prompt 猜测
- tool 权限具备 phase-aware 守门
- `active_skill` 锁定与新 phase 架构兼容
- trace、contract、eval 都能看见当前 phase 与路由证据
