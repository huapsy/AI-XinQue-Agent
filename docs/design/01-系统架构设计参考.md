## 一、总体架构思路：两部分系统（Two-Part System）

核心理念：**不直接让 LLM 做治疗决策**，而是用"临床理解模块 + LLM语言生成"两部分协作。

```
用户输入
  │
  ▼
┌─────────────────────────────────────────────┐
│  第一模块（First Module）— 目标导向的临床理解系统    │
│  ┌───────────────┐  ┌───────────────────┐     │
│  │ 主体理解模块    │  │ 推荐系统           │     │
│  │ (多个轻量模型)  │──▶│ (干预推荐/治疗路径) │     │
│  └───────┬───────┘  └────────┬──────────┘     │
│          │                   │                │
│          ▼                   ▼                │
│  ┌──────────────────────────────────┐         │
│  │ 提示生成模块                       │         │
│  │ (系统提示 = 模板 + 用户画像 + 知识) │         │
│  └──────────────┬───────────────────┘         │
└─────────────────┼───────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────┐
│  第二模块（Second Module）— 通用 LLM              │
│  接收系统提示 → 生成共情、个性化的自然语言回应        │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│  安全模块                                      │
│  输出安全检查 → 过滤有害内容 → 输出给用户           │
└─────────────────────────────────────────────┘
```

### 为什么不直接用 LLM

| 问题 | 说明 |
|------|------|
| 缺乏目标导向性 | LLM 通过模式匹配生成文本，无法像治疗师一样有长期治疗计划 |
| 缺乏可解释性 | 黑箱输出无法向临床人员解释"为什么给出这个回应" |
| 缺乏领域知识 | LLM 缺少真正的临床知识，可能产生有害幻觉 |
| 上下文窗口有限 | 无法维持跨会话的长期关系和患者理解 |
| 微调代价高 | 要让 LLM 学会新的临床技能需要大量数据和计算 |

## 二、各模块详细设计

### 2.1 输入处理

**文本输入**：直接进入处理管线。

**语音输入**：先经过 ASR（自动语音识别）转为文本，再进入处理管线。Azure OpenAI 可结合 Azure Speech Services 实现。

**数值输入**：如情绪评分量表（1-10分）、PHQ-9问卷分数等。

### 2.2 主体理解模块（Subject Understanding Module）

负责从用户输入中提取临床相关信息，输出"主体画像信息"(Subject Profile Information)。

包含多个**轻量级专用模型**（参数量仅数千到数万，远小于LLM的数百亿）：

| 子模型 | 输入 | 输出 | 用途 |
|--------|------|------|------|
| 认知扭曲检测模型 | 用户文本 | 是否扭曲 + 扭曲类型 + 概率值 | 判断是否需要认知重构 |
| 行为理解模型 | 用户文本 | 适应性/非适应性行为分类 | 识别不健康行为模式 |
| 情感分析模型 | 用户文本 | 情绪类型（悲伤/愤怒/焦虑/恐惧等） | 追踪情绪变化 |
| 情感检测模型 | 用户文本 | 正面/负面/中性 | 粗粒度情感判断 |
| 医学术语检测模型 | 用户文本 | 识别到的医学/心理学术语 | 理解用户专业表述 |
| 领域检测模型 | 用户文本 | 话题领域（人际/工作/健康/教育等） | 确定对话方向 |

**实现建议（基于 Azure OpenAI）**：
- 这些轻量模型可用 GPT 的 Function Calling / Structured Output 替代，即用一次 LLM 调用提取多个维度的结构化信息
- 也可以用传统 NLP 模型（如微调的 BERT）部署为独立服务
- 混合方案：关键分类用专用模型保证可靠性，辅助信息用 LLM 提取

### 2.3 知识库（Knowledge Bank）

存储治疗相关的结构化知识，类似 RAG 机制：

**知识库内容**：
- 临床治疗协议和指导方针
- 认知行为治疗（CBT）的具体技术说明
- 各类认知扭曲的定义、示例、应对方法
- 干预练习的脚本和步骤
- 危机干预的标准操作流程
- 患者背景信息（跨会话持久化）

**检索机制**：
1. 将用户输入文本通过语言模型生成嵌入向量
2. 与知识库中每条条目的嵌入向量计算余弦相似度
3. 选取相似度最高的 Top-K 条目（文献中推荐 K=5）
4. 将这些条目拼接到系统提示中，在用户输入之前

**实现建议**：
- Azure AI Search 作为向量检索引擎
- Azure OpenAI Embeddings API 生成嵌入
- 或使用 Azure Cosmos DB 的向量搜索功能

### 2.4 推荐系统（Recommender Module）

根据主体画像信息决定推荐什么干预措施。

**输入**：
- 主体画像信息（认知扭曲、情绪状态、行为模式等）
- 存储的诊断信息
- 用户人口统计信息
- 历史干预记录及用户反馈
- 当前对话上下文

**输出**：干预推荐（从预定义干预列表中选择）

**优化目标**（三个指标的平衡）：
1. **治疗联盟** — 干预是否有助于增强信任关系
2. **患者结局** — 干预是否有助于改善症状
3. **临床洞察** — 干预是否能帮助获取更多诊断信息

**决策逻辑**：
- 简单版：规则引擎（if-else），将主体画像映射到预定义干预列表
- 进阶版：训练推荐模型，基于历史数据学习最优推荐策略
- 阈值机制：当画像信息不充分时，指示交互模块继续收集信息，而非过早推荐

**预定义干预类型示例**：
- 认知重构练习
- 呼吸练习
- 正念冥想
- 行为激活
- 思维记录
- 情绪重新框架
- 渐进式肌肉放松

### 2.5 提示生成模块（Prompt Generation Module）

将所有信息组装成系统提示，送入 LLM。

**系统提示的组成**：
```
[角色定义] — 你是一个AI心理助手，使用心理治疗技术帮助用户...
[上下文说明] — 你正在与用户进行对话，像心理治疗师在会话中那样...
[当前目标] — 基于主体画像的当前治疗目标和推荐的干预
[临床标注] — 例如 "### 扭曲思维 ###" 或 "### 非扭曲思维 ###"
[知识库检索结果] — RAG 返回的相关知识条目
[对话历史] — 之前的对话轮次
[用户当前输入] — 最新的用户消息
```

**提示模板可以有多个**，根据第一模块的输出选择不同模板：
- 通用对话模板
- 认知重构专用模板
- 危机干预模板
- 信息收集模板
- 干预介绍/邀请模板

### 2.6 输出安全模块（Output Safety Module）

在 LLM 输出到达用户之前进行安全检查。

**检查方式**：
1. **关键词/正则匹配**：扫描预定义的危险词汇和短语
2. **训练分类器**：前馈神经网络，将输出文本分类为"安全"/"不安全"
3. **LLM 自检**：用另一个 LLM 调用判断输出是否偏离主题或有害

**不安全时的处理**：
- 阻止输出到达用户
- 修改系统提示后重新生成
- 切换到规则引擎（rule-based）生成安全回应
- 转介人类治疗师（发送邮件/短信/电话提醒）

### 2.7 输入安全模块（Subject Safety Module）

在生成回应之前评估用户输入。

**检测内容**：
- 自杀意念或自我伤害的表述
- 对他人的暴力威胁
- 虐待儿童相关内容
- 恐慌发作的语言特征
- 创伤性事件披露
- 提示注入攻击

**危机处理流程**：
- 检测到危机 → 跳过正常对话流程
- 直接提供危机干预脚本（预定义的安全回应）
- 提供紧急联系方式（如心理危机热线）
- 通知人类治疗师或紧急服务

## 三、Wysa 的对话对齐架构补充

Wysa 的架构更侧重对话过程管理：

```
用户消息
  │
  ├──▶ 上下文收集模块 ──▶ 危机情况检测
  │    (情绪/医学术语/领域)
  │
  ├──▶ 情感检测模块（正/负/中）
  │
  ▼
对话对齐监控模块
  ├── 对齐检测（用户在配合）→ 加分
  ├── 不对齐检测（用户抗拒/困惑）→ 减分 → 生成恢复提示
  │
  ▼
对齐分数监控
  │
  ├── 未达阈值 → 继续对话、收集上下文
  ├── 达到阈值 → 联盟确认提示 → 用户同意 → 开始干预
  │
  ▼
提示生成模块 → LLM → 输出给用户
```

## 四、数据流总结

```
用户输入
  │
  ▼
[输入安全检查] ──危机──▶ 危机干预流程
  │
  ▼（安全）
[主体理解模块] → 输出主体画像（认知扭曲/情绪/行为/领域）
  │
  ├──▶ [知识库检索] → 返回相关知识条目
  ├──▶ [对话对齐评估] → 更新对齐分数
  ├──▶ [推荐系统] → 推荐干预措施（若对齐分数达标）
  │
  ▼
[提示生成模块] → 组装系统提示
  │
  ▼
[LLM (GPT 5.x)] → 生成回应
  │
  ▼
[输出安全检查] ──不安全──▶ 重新生成 / 规则回应 / 转介人工
  │
  ▼（安全）
输出给用户
  │
  ▼
[记录与持久化] → 更新用户画像 / 对话历史 / 治疗进度
```

## 五、与 Azure OpenAI 的映射

| 架构组件 | Azure 实现方案 |
|----------|--------------|
| LLM（第二模块） | Azure OpenAI GPT-5.4/5.3-codex |
| 主体理解模块 | GPT Structured Output / Function Calling 提取多维信息 |
| 知识库 + RAG | Azure AI Search + Azure OpenAI Embeddings |
| 输入/输出安全 | Azure AI Content Safety + 自定义规则 |
| 语音识别 | Azure Speech Services STT |
| 语音合成 | Azure Speech Services TTS |
| 数据存储 | Azure Cosmos DB（用户画像/对话历史） |
| 向量存储 | Azure AI Search 向量索引 / Cosmos DB 向量搜索 |
