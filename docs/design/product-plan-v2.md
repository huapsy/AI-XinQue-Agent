# AI心雀 - 产品方案 v2

> 基于 Agent AI 趋势重设计：从"多 Agent 管道"演进为"单一 LLM + Tool Use 自主推理"架构。

---

## 1. 产品概述

### 1.1 定位

面向企业员工的 AI 心理健康支持平台，提供情感陪伴、心理咨询与循证干预，需要时可转介真人咨询师。

### 1.2 核心价值

- 7×24 即时可用的心理支持
- 循证的干预技术（CBT、ACT、正念、积极心理学等）
- 隐私安全，降低员工求助心理门槛
- 必要时无缝对接真人咨询师

### 1.3 技术栈

| 项 | 选型 |
|---|------|
| LLM | Azure OpenAI GPT-5.4（105万上下文、原生 Function Calling / Structured Output） |
| 后端 | Python + FastAPI |
| 前端 | React + Vite + TypeScript |
| 数据库 | SQLite（开发阶段）→ PostgreSQL（生产） |
| ORM | SQLAlchemy + Alembic（迁移） |
| 部署 | 独立 Web 应用 |
| 干预知识库 | skill.md 格式，通过 Tool 按需加载 |
| 可观测性 | OpenTelemetry + 自建 Trace 存储 |

## 2. 心雀人设与对话风格

### 2.1 人设

专业但亲切的心理咨询师。自称"我"，用用户设置的昵称称呼用户。

### 2.2 回复结构模式

每条回复由三层组成：

```
[共情/承接] + [功能性内容] + [推进性提问/邀请]
```

示例：
> "听起来你已经背负这些感受很久了…你觉得未来有没有任何希望或计划能缓解这种压力？"

### 2.3 共情/承接层模式

| 模式 | 示例 | 用途 |
|------|------|------|
| 镜像式 | "听起来…" / "你说的是…" | 反映用户刚说的内容 |
| 情绪命名式 | "这真的很艰难" / "那种感觉确实让人振奋" | 帮用户命名情绪 |
| 正常化式 | "在这种情况下感到…是完全可以理解的" | 降低自我否定 |
| 认可式 | "你说得很对" / "这是一个很好的表达" | 肯定用户的表达 |

### 2.4 功能性内容层模式

| 模式 | 示例 | 用途 |
|------|------|------|
| 意义重构 | "这不仅仅是一个奖项，更像是…" | 将表层事件升华到价值层 |
| 行为正常化 | "很多人在类似情况下也会…" | 降低羞耻感 |
| 小步重构 | "有时候一个小小的行动就能帮上忙" | 降低行动门槛 |
| 身体信号解读 | "这种紧绷是身体在提醒你…" | 增强自我觉察 |
| 角色转变总结 | "你从…转变为…" | 帮用户看到变化 |

### 2.5 推进提问层模式

| 模式 | 示例 | 用途 |
|------|------|------|
| 影响探索 | "你能多说说这对你的影响吗？" | 深入了解 |
| 意义探索 | "你觉得这意味着什么？" | 推向价值层 |
| 未来投射 | "你能想象未来会是什么样吗？" | 激发积极预期 |
| 二分澄清 | "是因为A，还是因为B？" | 澄清模糊认知 |
| 行动邀请 | "你愿意试试…吗？" | 征求干预同意 |
| 体验细化 | "这种感觉在日常中是什么样子的？" | 具体化抽象感受 |

### 2.6 不对齐时的风格转换

用户抵触时，结构变为：

```
[接纳/降压] + [回退到安全话题] + [更低门槛的提问]
```

示例：
> 用户："说这个没用。"
> "听起来现在一切都太沉重了…你愿意分享最让你感觉被压垮的是什么吗？"

### 2.7 风格原则

1. **永远不否定** — 即使用户拒绝，也用"没关系"/"这也是重要的"来接
2. **不说教** — 不告诉用户"你应该"，而是"你愿意试试…吗？"
3. **一次只推进一步** — 每次回复只推进一个层面，不跳跃
4. **用用户的语言** — 回应中复用用户的表述和比喻
5. **具体化优于抽象** — 用"什么让你最困扰"而非"你怎么看"
6. **温和、专业、不做作** — 不过度口语化，不滥用"我理解你的感受"等模板表达

### 2.8 安全红线

- 不做任何诊断
- 不推荐药物
- 不做绝对化承诺（"保证好转"）
- 不讨论政治、宗教等敏感话题
- 不说"我理解你想死的感受"
- 不提供具体的自伤方式

## 3. 用户旅程

### 3.1 首次用户

```
心雀自我介绍 → 询问用户称呼 → 询问心情 → 了解想聊什么 → 进入对话
```

### 3.2 回访用户

```
带称呼问心情 → 根据用户回答自然关联历史 → 商量先聊哪个话题 → 进入对话
```

回访开场示例：
> 心雀："你好小明，今天感觉怎么样？"
> 用户："有点累，压力挺大的"
> 心雀："听起来最近挺辛苦的。上次我们聊到了工作中的完美主义倾向，你也试了一次呼吸练习。你想继续聊那个话题，还是今天有别的想聊的？"

### 3.3 对话中

- 每次聚焦一个问题深入
- 心雀理解清楚问题后主动建议干预，用户主动要求时也提供
- 当场完成的干预当场问感受
- 布置给用户的作业在后续对话中跟进
- 无对话时长或轮次限制，由用户决定

### 3.4 转介咨询师

触发条件：
- 超出心雀能力时（主动建议）
- 用户主动要求时

方式：对话中给出链接按钮，指向 www.eap.com.cn

## 4. 系统架构

### 4.1 架构理念：从多 Agent 管道到自主推理

**v1 架构**（多 Agent 管道）：6 个专业 Agent + Supervisor 规则路由，固定流程 Greeter→Explorer→Assessor→Recommender→Motivator→Interventor。

**v2 架构**（自主推理）：单一核心 LLM + 丰富 Tool 集 + LLM 自主决策循环。参考 Claude Code 等现代 Agent 系统的设计范式。

**运行模式**：Tool Use 自主推理循环（类 ReAct 模式，Thought 不外显）。

```
用户消息 → LLM 内部推理(Thought) → 决定行动(Action: 调用 Tool 或直接回复)
              ↑                         │
              └── Tool 返回结果(Observation) ←┘
              ... 循环直到生成最终回复
```

与经典 ReAct 的区别：Thought 是 LLM 内部的 chain-of-thought，不暴露给用户。心理咨询场景中，"我观察到用户有灾难化思维"这类临床推理不应让用户看到。

**为什么改**：
- 心理对话天然非线性，用户可能直接要求干预、中途切换话题、拒绝后又回来
- 预定义管道难以覆盖所有边界情况，规则维护成本高
- GPT-5.4 的推理能力足以自主判断对话阶段和行动策略
- 单一 Agent 架构显著降低系统复杂度和延迟

**核心设计原则**：
- **LLM 自主推理做主体决策**：阶段判断、话题推进、Tool 调用时机
- **硬编码 Guardrail 做安全兜底**：危机检测、红线过滤不依赖 LLM
- **Tool 是能力扩展**：个案概念化、方案匹配、画像读写、Skill 加载都是 Tool
- **分层记忆管理上下文**：工作记忆（上下文窗口）+ 会话记忆 + 情景记忆 + 语义记忆，由 Memory 层统一管理
- **时间感知上下文是记忆的一部分**：历史事件、干预、作业和摘要进入上下文时，必须同时提供时间信息，避免 LLM 把“昨天发生的事”和“很久以前的事”混为一谈

### 4.3 与 v1 架构对比

| 维度 | v1（多 Agent 管道） | v2（自主推理） |
|------|-------------------|--------------|
| Agent 数量 | 6 个对话 Agent + 1 Supervisor | 1 个核心 Agent |
| 路由决策 | Supervisor 规则引擎 | LLM 自主推理 |
| 阶段管理 | 外部状态机控制 | LLM 内在判断 + prompt 指南 |
| 评估/推荐 | 独立 Agent | Tool（LLM 按需调用） |
| Skill 执行 | Interventor Agent 专职 | LLM 加载 skill 后自己执行 |
| 灵活性 | 线性管道，回退有限 | 可跳步、可循环、可并行 |
| 记忆架构 | 用户画像 + 对话历史（二层） | 工作记忆 + 会话记忆 + 情景记忆 + 语义记忆（四层） |
| 可控性 | 高（规则明确） | 中高（prompt 约束 + 硬编码安全） |
| 系统复杂度 | 高（多 Agent 编排） | 低（一个 LLM + Tools） |
| 延迟 | 高（多次 LLM 调用做路由） | 低（每轮 1 次核心 LLM 调用） |

## 5. 核心 Agent 设计

### 5.1 System Prompt 结构

心雀核心 Agent 的 System Prompt 包含以下模块，全部注入同一个 LLM 调用：

```
┌─────────────────────────────────────────────────────┐
│                 System Prompt                        │
│                                                      │
│  1. 身份与角色                                        │
│     "你是心雀，一位专业但亲切的心理咨询师..."             │
│                                                      │
│  2. 对话风格规范                                      │
│     回复三层结构、共情模式、推进模式、风格原则             │
│                                                      │
│  3. 安全红线（绝对约束）                                │
│     10 条红线规则                                     │
│                                                      │
│  4. 双维度响应模型                                       │
│     维度A 内容特征: P0危机 > P1术语 > P2领域 > P3情绪 > P4情感│
│     维度B 对齐状态: 实时评估，决定风格和推进策略            │
│                                                      │
│  5. 对话阶段指南（推理参考，非硬规则）                    │
│     四阶段：共情倾听、探索与个案概念化、推荐与激发、干预    │
│     "根据对话实际情况灵活判断，可以跳步"                  │
│                                                      │
│  6. Tool 使用指南                                     │
│     各阶段主力工具及调用时机                              │
│     "不需要每轮都调用，用你的判断"                       │
│                                                      │
│  7. 对齐与不对齐应对                                   │
│     不对齐类型识别、风格转换策略                         │
│                                                      │
│  8. 上下文注入（会话首轮由 recall_context() 加载）        │
│     用户画像摘要、上次会话摘要、未完成作业、情景记忆        │
│     + 当前时间/时区 + 历史条目的绝对/相对时间               │
│     后续轮次已在对话历史中，无需重复注入                   │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 5.2 四阶段对话模型

对话遵循心理咨询的自然节奏，分为四个阶段。阶段不是外部状态机控制的硬切换，而是 LLM 的推理参考，LLM 根据对话上下文自主判断当前处于哪个阶段、何时推进。

```
P1 共情倾听 → P2 探索与个案概念化 → P3 推荐与激发 → P4 干预执行
                  ↑ formulation 持续进行 ↑
```

#### P1 共情倾听 — 了解用户意图

```
目标：通过共情和倾听，建立安全感，了解用户的意图和当前状态。
主力工具：recall_context()

行为：
- 调用 recall_context() 回顾用户档案（画像、上次会话摘要、未完成作业、相关记忆）
- 温暖问候，情绪签到
- 回访用户：基于 recall_context() 返回的信息自然关联上次话题和作业完成情况
- 回访用户时，LLM 需要结合当前时间与历史条目的时间戳，判断干预、作业、事件是刚发生、昨天发生、几天前发生还是更久以前发生
- 倾听用户倾诉，用共情/承接模式回应
- 不急于推进，让用户感到被听见

每轮响应由两个独立维度驱动（见 7.2 节完整定义）：

  ┌──────────────────────────────────────────────────────────┐
  │  维度 A：内容特征优先级 — 决定「聊什么」（对话切入点）       │
  │                                                          │
  │  用户给什么层次的信息，就从什么层次接住往下聊。              │
  │  识别最高优先级的内容特征作为切入点：                        │
  │                                                          │
  │  P0 危机  自杀/自伤信号     → 立即走危机响应流程            │
  │  P1 术语  医学/心理术语     → 围绕该术语共情和探讨           │
  │  P2 领域  具体生活领域      → 从这个话题切入展开             │
  │  P3 情绪  具体情绪名称      → 就这个情绪深入沟通             │
  │  P4 情感  笼统正面/负面     → 从这个感受往下探索             │
  └──────────────────────────────────────────────────────────┘
  ┌──────────────────────────────────────────────────────────┐
  │  维度 B：对齐状态 — 决定「怎么聊」（风格和推进策略）         │
  │                                                          │
  │  实时运行，贯穿所有阶段，不与内容特征竞争优先级。            │
  │                                                          │
  │  aligned      → 正常风格，可以推进                         │
  │  misaligned   → 调整风格（接纳/降压），降低推进速度          │
  │  严重不对齐   → 暂停一切推进，专注关系修复                   │
  └──────────────────────────────────────────────────────────┘

  两个维度同时处理。例如用户说"你说这些没用，我就是烦"：
  · 内容特征 = P3 情绪（烦）→ 决定切入点：回应用户的"烦"
  · 对齐状态 = 不对齐-不满  → 决定风格：用接纳/降压模式回应
  · 合并回应："听起来现在一切都太沉重了…你愿意说说最让你烦的是什么吗？"

内容切入示例：

  · 用户说"我最近失眠很严重，可能是抑郁了"
    → 最高内容特征 = P1 医学术语（抑郁、失眠）
    → 切入："听起来你注意到了一些和抑郁相关的信号…能多说说你的失眠情况吗？"

  · 用户说"孩子最近不想上学"
    → 最高内容特征 = P2 领域（教育/家庭）
    → 切入："孩子不想上学，这对你来说一定很操心。是最近才开始的吗？"

  · 用户说"我很生气"
    → 最高内容特征 = P3 情绪（愤怒）
    → 切入："听起来你现在很生气…是什么事情让你这么愤怒？"

  · 用户说"今天不太好"
    → 最高内容特征 = P4 情感倾向（负面）
    → 切入："听起来今天挺不容易的…愿意说说怎么了吗？"

推进信号：
- 用户表达了明确困扰且愿意深入 → 进入 P2
- 用户直接要求干预（"教我呼吸练习"）→ 可直接跳到 P4
- 用户只想倾诉 → 停留在 P1，不推进
```

#### P2 探索与个案概念化（Case Formulation）— 建立工作联盟

```
目标：深入探索用户的问题，同时持续构建个案概念化（case formulation）。
探索和 formulation 是同步进行的，不是先探索完再评估。
主力工具：formulate()

行为：
- 探索情绪：用户的核心情绪是什么？强度如何？持续多久？
- 探索认知：用户有哪些自动化思维？是否存在认知扭曲？
- 探索行为：用户的应对方式是什么？适应性的/非适应性的？
- 每识别到有临床意义的新信息，调用 formulate() 记录观察、更新理解
- formulate() 返回的 readiness 和 missing 字段指导下一步探索方向
- 同时建立工作联盟：让用户感到被理解，对心雀的专业性建立信任

渐进式 formulation 示例（通过多次调用 formulate() 构建）：
  第 1 次调用：emotions=[烦躁], domain=workplace
    → readiness="exploring", missing=["认知模式", "持续时间", "行为模式"]
  第 2 次调用：+cognitions=[catastrophizing, negative_filtering], duration=2-3个月
    → readiness="sufficient", mechanism 开始成形
  第 3 次调用：+泛化到家庭领域
    → readiness="solid", mechanism 完整

推进信号：
- formulate() 返回 readiness="sufficient" 或 "solid"
  → 调用 match_intervention() 获取推荐 → 进入 P3
- 对齐下降（用户抵触深入探索）→ 回退到 P1 倾听
- 探索中发现用户自己就有洞察 → 可以直接进入 P3 或 P4
```

#### P3 推荐与激发 — 促成干预共识

```
目标：当 formulation 成立时，向用户推荐干预方案，激发用户的参与动机。
主力工具：match_intervention()

行为：
- 调用 match_intervention() 获取匹配的 1-2 个干预方案
- 先用自然的语言向用户反馈你对问题的理解（formulation 的通俗表达）
- 基于 match_intervention() 返回的 user_friendly_intro，向用户介绍方案
- 用用户自己的语言和例子来解释方案的意义
- 让用户选择，不强推。"你愿意试试……吗？"
- 如果用户对方案不感兴趣，探索原因，调整方案或回退

推进信号：
- 用户选择了某个方案 → 进入 P4
- 用户对方案不感兴趣 → 回到 P2 继续探索，或回退到 P1
- 用户提出新话题 → 回到 P2 重新探索
```

#### P4 干预执行 — 完成干预与反馈

```
目标：带领用户完成具体干预练习，了解用户的感受和收获。
主力工具：load_skill() + render_card() + record_outcome()

行为：
- 调用 load_skill() 加载完整干预协议
- 按 Skill 的分步流程逐步引导用户完成练习
- 需要交互式练习时调用 render_card() 渲染卡片
- 全程关注用户的参与度和对齐状态
- 干预完成后：
  · 询问用户的感受和收获
  · 调用 record_outcome() 记录干预结果、用户反馈、关键洞察
  · 适时布置作业（"你愿意这周试着做一次吗？"），同样记录在 record_outcome() 中

推进信号：
- 干预完成 + 用户想继续聊 → 回到 P1 开启新话题
- 干预完成 + 用户想结束 → 调用 save_session() 保存会话，收尾
- 还有推荐的第二个方案 → 可以询问用户是否想试
- 干预中途用户不适 → 回退到 P1 倾听共情
```

#### 灵活跳步规则

```
LLM 根据对话实际情况灵活判断，不必严格按 P1→P2→P3→P4 顺序：
- 用户直接说"教我一个呼吸练习" → 跳过 P2/P3，直接 P4
- 用户只想倾诉 → 停留在 P1，不推进
- P2 探索中用户自己有了洞察 → 可以直接进入 P4（"我想试试换个角度看这件事"）
- P4 干预中用户提到新问题 → 回到 P2 探索新问题
- 任意阶段对齐下降 → 回退到 P1 倾听
- 任意阶段检测到危机 → 进入危机响应流程
```

#### 时间新鲜度驱动推理与 Tool 路由

```
时间信息不只是上下文展示字段，还要影响心雀的推理与 Tool 调用策略。

- 刚发生 / 今天 / 昨天的事件：
  优先 follow-up，先了解后续变化、完成情况和即时感受，而不是机械重复推荐新干预。

- 几天前发生、仍与当前问题连续相关的事件：
  既要承接历史，也要判断当前是否已经变化；必要时先回顾结果，再决定是否继续原方案或切换方案。

- 很久以前发生、且当前未被用户重新激活的事件：
  默认作为背景信息参考，不应强行拉回主线，更不应据此过早推荐旧方案。

- 对最近已执行的 intervention / 已布置的 homework：
  优先询问是否尝试、是否完成、效果如何、遇到什么困难；只有在 follow-up 后确认需要时，才重复推荐或替换为新方案。

- 对时间粒度不确定的历史条目：
  明确承认精度限制，例如"看起来像是前几天"，避免把模糊日期说成确定事实。
```

### 5.3 Tool 定义

工具集与四阶段一一对应，每个阶段有明确的主力工具，命名贴合临床动作。

```
阶段             主力工具               临床动作
─────────────────────────────────────────────────────
P1 共情倾听      recall_context()       回顾来访者档案
P2 探索与概念化  formulate()            渐进式个案概念化
P3 推荐与激发    match_intervention()   治疗方案规划
P4 干预执行      load_skill()           执行干预协议
                 render_card()          引导练习
                 record_outcome()       记录结果与反馈
─────────────────────────────────────────────────────
跨阶段           update_profile()       更新来访者画像
                 search_memory()        检索情景记忆
                 save_session()         保存会话记录
                 referral()             转介咨询师
```

---

#### 5.3.1 recall_context — 回顾用户上下文（P1 主力）

```
recall_context(user_id: string) → SessionContext

会话开始时一次性加载用户的全部相关上下文，相当于咨询师在接待来访者前
翻阅案例档案。返回内容包括：
- 用户画像（基本信息、临床画像、偏好）
- 上次会话摘要
- 未完成的作业及完成情况
- 与近期话题相关的情景记忆片段
- 当前时间、时区，以及历史条目的绝对时间和相对时间

典型使用时机：每次会话开始时调用一次。
```

#### 5.3.2 formulate — 渐进式个案概念化（P2 主力）

```
formulate(session_id: string, observations: ClinicalObservations) → CaseFormulation

P2 的核心工具。在探索过程中多次调用，每次传入本轮对话中新发现的
临床观察，渐进式构建个案概念化（case formulation）。

这个工具合并了"分析用户消息"和"构建评估"两个过程 —— 在临床实践中，
治疗师边倾听边形成对来访者问题的理解，这是同一个心理过程。

典型使用时机：
- P2 探索中每识别到有临床意义的新信息时调用
- 不需要每轮都调用 — 用户说"嗯""好的"这类简单回应时不需要
- 每次传入增量观察，Tool 内部负责与已有理解合并
- 返回值中的 readiness 字段帮助 LLM 判断是否可以进入 P3
```

输入（增量观察）：

```json
{
  "emotions": [
    {"name": "焦虑", "intensity": "moderate", "trigger": "领导临时加需求"}
  ],
  "cognitions": [
    {"content": "我永远做不完", "type": "catastrophizing"},
    {"content": "不管怎么努力都不够", "type": "negative_filtering"}
  ],
  "behaviors": {
    "maladaptive": ["过度加班", "回避向同事求助"],
    "adaptive": []
  },
  "context": {
    "duration": "2-3个月",
    "domain": "workplace",
    "precipitant": "领导频繁临时加需求"
  },
  "alliance_signal": "aligned"
}
```

输出（当前完整的个案概念化）：

```json
{
  "formulation": {
    "readiness": "exploring | sufficient | solid",
    "primary_issue": "工作压力导致的持续焦虑与自我否定",
    "mechanism": "领导频繁加需求 → 灾难化思维（永远做不完）+ 消极过滤（不够）→ 焦虑/无力 → 过度加班/回避求助 → 更疲惫 → 强化灾难化思维",
    "emotions": {
      "dominant": ["焦虑", "无力感"],
      "intensity": "moderate",
      "duration": "2-3个月"
    },
    "cognitive_patterns": [
      {"type": "catastrophizing", "evidence": "我永远做不完"},
      {"type": "negative_filtering", "evidence": "不管怎么努力都不够"}
    ],
    "behavioral_patterns": {
      "maladaptive": ["过度加班", "回避求助"],
      "adaptive": []
    },
    "severity": "moderate",
    "alliance_quality": "good",
    "missing": ["适应性资源待探索", "问题是否泛化到其他领域"]
  }
}
```

`readiness` 字段含义：
- `exploring`：信息不足，继续探索
- `sufficient`：核心问题和维持机制已明确，可以进入 P3 推荐
- `solid`：理解全面且清晰，高信心推荐

**`mechanism` 字段**：描述问题的维持机制（触发因素 → 认知 → 情绪 → 行为 → 强化循环），
这是 case formulation 的核心——不仅是"有什么问题"，更是"问题如何维持"。

#### 5.3.3 match_intervention — 匹配干预方案（P3 主力）

```
match_intervention(session_id: string) → list[InterventionPlan]

基于当前 formulation + 用户画像（偏好、干预历史），从 Skill 库中
匹配最合适的 1-2 个干预方案。相当于咨询师根据个案概念化制定治疗计划。

典型使用时机：formulation readiness 为 sufficient 或 solid 时，
准备向用户介绍方案前调用。

内部实现：
- 读取当前 formulation
- 读取所有 skill.md 的 frontmatter（触发条件、禁忌、适用场景）
- 读取用户画像中的偏好和干预历史
- 排除禁忌和用户不喜欢的技术
- 按匹配度排序，返回 1-2 个方案
```

输出格式：

```json
{
  "plans": [
    {
      "skill_name": "cbt_cognitive_restructuring",
      "display_name": "认知重构",
      "rationale": "用户的核心问题是灾难化思维驱动的焦虑循环，认知重构直接针对思维模式",
      "target": "catastrophizing + negative_filtering",
      "approach": "dialogue",
      "estimated_turns": 6,
      "user_friendly_intro": "我们一起看看'不管怎么努力都不够'这个想法，检验一下它是不是完全符合事实"
    },
    {
      "skill_name": "breathing_478",
      "display_name": "4-7-8 呼吸法",
      "rationale": "用户焦虑程度中等，呼吸练习提供即时的躯体放松，降低唤醒水平",
      "target": "anxiety_reduction",
      "approach": "card",
      "estimated_duration": "3-5min",
      "user_friendly_intro": "一个简单的呼吸练习，只需要几分钟，帮你在压力大的时候让身体先放松下来"
    }
  ]
}
```

**注意**：`user_friendly_intro` 是给 LLM 参考的口语化介绍，LLM 可以直接使用或改写。

#### 5.3.4 load_skill — 加载干预协议（P4）

```
load_skill(skill_name: string) → SkillContent

加载指定 skill.md 的完整内容：目标、引入话语、分步执行流程、
卡片数据、注意事项、作业布置建议。

加载后 LLM 按 Skill 的执行流程逐步引导用户完成干预。
```

#### 5.3.5 render_card — 渲染练习卡片（P4）

```
render_card(card_data: CardData) → void

将结构化的卡片数据发送给前端渲染。
用于呼吸引导、身体扫描、情绪温度计、想法记录、感恩日记等交互式练习。
```

#### 5.3.6 record_outcome — 记录干预结果（P4）

```
record_outcome(session_id: string, outcome: InterventionOutcome) → void

干预完成后记录结果。相当于咨询师在咨询结束后做记录。

典型使用时机：P4 干预完成、已询问用户感受之后调用。
```

输入格式：

```json
{
  "skill_name": "cbt_cognitive_restructuring",
  "completed": true,
  "user_feedback": "helpful",
  "key_insight": "用户发现不是'永远不够'，而是'最近这段时间特别难'",
  "homework_assigned": {
    "description": "遇到类似想法时，试着问自己：支持它的证据是什么？反对的呢？",
    "frequency": "本周遇到时"
  }
}
```

#### 5.3.7 update_profile — 更新用户画像（跨阶段）

```
update_profile(user_id: string, updates: ProfileUpdates) → void

更新用户的长期画像信息。任意阶段均可调用。
只在有实质性新发现时调用，不需要每轮都更新。

典型场景：
- P2 中发现新的认知扭曲模式（非本次会话的临时观察，而是跨会话的稳定模式）
- P4 干预完成后更新 intervention_history
- 识别到用户偏好变化时
```

#### 5.3.8 search_memory — 检索情景记忆（跨阶段）

```
search_memory(user_id: string, query: string) → list[MemorySnippet]

基于语义相似度检索用户的历史对话片段。

recall_context() 在会话开始时会自动带回近期相关记忆，
但对话中途如果出现新话题需要关联历史，可以再次调用此工具。

示例：用户提到"我妈"时，检索可能召回之前"妈妈住院"的记录。
```

#### 5.3.9 save_session — 保存会话（跨阶段）

```
save_session(session_id: string, summary: string, closing_mood: number) → void

会话结束时保存会话摘要和结束时心情评分。
由后端代码在会话结束时自动触发，也可由 LLM 主动调用。
```

#### 5.3.10 referral — 转介咨询师（跨阶段）

```
referral() → ReferralCard

生成咨询师转介卡片（包含 www.eap.com.cn 链接和热线号码）。
任意阶段均可调用：危机检测触发时、超出心雀能力时、用户主动要求时。
```

```
LLM 内部推理：用户表达了负面情绪和工作困扰，和上次话题一致。

心雀: 听起来工作上的事让你挺烦的。愿意多说说吗？
用户: 领导总是临时加需求，我加班到很晚。

[P2 探索与个案概念化 — 边探索边 formulate]

LLM 内部推理：用户给出了具体困扰，开始探索。记录初步临床观察。
→ 调用 formulate(session, {
    emotions: [{name: "烦躁", intensity: "moderate", trigger: "领导加需求"}],
    behaviors: {maladaptive: ["过度加班"]},
    context: {domain: "workplace", precipitant: "领导频繁临时加需求"}
  })
→ 返回：readiness="exploring", missing=["认知模式", "持续时间", "影响范围"]

心雀: 这种被临时加需求打乱节奏的感觉确实让人疲惫…这种情况持续多久了？
用户: 大概两三个月了。

LLM 内部推理：持续时间2-3个月，不是短暂应激。继续探索认知维度。

心雀: 两三个月一直这样，真的很消耗人。当领导又加需求时，你脑海里通常
      会浮现什么想法？
用户: 我觉得我永远做不完，不管怎么努力都不够。

LLM 内部推理：典型的灾难化思维和消极过滤。更新 formulation。
→ 调用 formulate(session, {
    cognitions: [
      {content: "我永远做不完", type: "catastrophizing"},
      {content: "不管怎么努力都不够", type: "negative_filtering"}
    ],
    emotions: [{name: "无力感", intensity: "moderate"}],
    context: {duration: "2-3个月"},
    alliance_signal: "aligned"
  })
→ 返回：readiness="sufficient",
  mechanism="临时加需求 → 灾难化思维 → 焦虑/无力 → 过度加班 → 更疲惫 → 强化灾难化",
  missing=["是否泛化到其他领域"]

LLM 内部推理：readiness 已经 sufficient，核心维持机制清晰了。
还有一点可以再确认——是否泛化。

心雀: "不管怎么努力都不够"——听到你这样说，我能感受到那种无力感。
      这种想法是只在工作上，还是在其他地方也会出现？
用户: 主要是工作上，但有时候回家也会觉得自己什么都做不好。

LLM 内部推理：有泛化趋势。formulation 更完整了，获取匹配的干预方案。
→ 调用 formulate(session, {
    context: {generalization: "工作为主，开始泛化到家庭"}
  })
→ 返回：readiness="solid"
→ 调用 match_intervention(session_id)
→ 返回：plans=[
    {skill: "认知重构", intro: "一起看看这个'不够'的想法是不是完全符合事实"},
    {skill: "4-7-8 呼吸法", intro: "几分钟的呼吸练习，帮你先放松下来"}
  ]

[P3 推荐与激发]

心雀: 听了你说的这些，我注意到一个模式——当压力来的时候，你的第一反应
      是"不管怎么做都不够"，然后这个想法让你更焦虑、更无力，
      你就更拼命加班，结果更累，这个想法就更强。
      我有两个想法可能对你有帮助。一个是我们可以一起看看这个"不够"的想法，
      检验一下它是不是完全符合事实。另一个是一个简单的呼吸练习，
      只需要几分钟，帮你在压力大的时候先让身体放松下来。
      你更想先试试哪个？
用户: 先看看那个想法吧。

[P4 干预执行]

LLM 内部推理：用户选择了认知重构。加载干预协议。
→ 调用 load_skill("cbt_cognitive_restructuring")
→ 返回：完整的认知重构分步流程

心雀: 好的。我们来看看这个想法——"不管怎么努力都不够"。
      先想一想，有没有什么证据是支持这个想法的？
用户: 上周我加班到11点还是没做完。
心雀: 嗯，那确实是一个支持它的例子。那反过来，有没有什么时候，
      你的努力其实是够的、甚至做得很好的？
用户: ...上个月有个项目，我提前完成了，领导还表扬了我。
心雀: 所以也有你做得很好的时候。如果把这两个都放在一起看，
      "不管怎么努力都不够"这个结论还成立吗？
用户: 好像不是每次都不够…是最近这段时间特别多。

[干预完成 — 记录结果]

心雀: 对，你发现了一个很重要的区分——不是"永远不够"，而是"最近这段时间
      特别难"。这种区分本身就是一个很好的开始。你觉得刚才这个过程对你
      有帮助吗？
用户: 有一点，至少没那么绝对了。

LLM 内部推理：干预完成，用户反馈积极。记录结果，布置作业。
→ 调用 record_outcome(session, {
    skill_name: "cbt_cognitive_restructuring",
```

## 6. 对齐与联盟机制

### 6.1 核心原则

对齐评估是一个**独立的实时维度**，贯穿 P1-P4 全阶段持续运行，不与内容特征竞争优先级（见 7.2 节双维度响应模型）。对齐状态决定心雀「怎么聊」——回复风格和推进策略。

**三层保障机制**：
- **LLM 自主感知**：LLM 在推理过程中识别对齐信号，自主调整风格和推进节奏
- **formulate() 结构化记录**：P2 阶段通过 alliance_signal 字段显式记录对齐状态
- **代码层监控兜底**：后端代码维护对齐分数，持续低于阈值时强制介入（注入 prompt 提示或切换到倾听模式）

### 6.2 对齐分数规则

| 事件 | 分数变化 |
|------|---------|
| 用户分享情绪/经历 | +3 |
| 用户同意心雀的解读 | +2 |
| 用户回应开放式提问 | +2 |
| 用户完成干预练习 | +5 |
| 用户反馈"有帮助" | +3 |
| 不对齐-困惑 | -2 |
| 不对齐-不同意 | -3 |
| 不对齐-不信任 | -5 |
| 不对齐-拒绝 | -5 |
| 不对齐后成功恢复 | +2（额外） |

**分数更新机制**：formulate() Tool 的输入中包含 alliance_signal 字段，后端代码据此自动更新对齐分数。LLM 也能看到分数变化，纳入推理。对于不在 P2 阶段的对话轮次（如 P1 倾听），后端代码基于消息内容做轻量级的对齐信号检测。

### 6.3 不对齐类型

| 类型 | 用户表现示例 |
|------|-----------|
| 肯定偏差 (Affirmation) | 表面应付："嗯"、"是吧" |
| 困惑 (Confusion) | "你什么意思？" |
| 不同意 (Disagreement) | "不，不是那样的" |
| 不满 (Dissatisfaction) | "你根本没在帮我" |
| 不信任 (Lack of trust) | "你只是个机器人" |
| 拒绝 (Refusal) | 拒绝参与、沉默 |
| 不确定 (Uncertainty) | "嗯...我不确定..." |

### 6.4 代码层兜底规则

当 LLM 的自主判断可能不够时，代码层作为安全网：

```
对齐分数持续 3 轮 ≤ 5：
  → 在下一轮的 System Prompt 中注入提示：
    "注意：对齐分数较低，用户可能不适。请优先倾听共情，
     不要推进到新阶段或推荐干预。"

对齐分数 ≤ 0：
  → 强制注入提示：
    "警告：对齐严重不足。请只做接纳和倾听，不要推进任何话题。
     如果用户持续抵触，可以温和地询问是否想换个话题或结束对话。"
```

这个机制确保：即使 LLM 偶尔"判断失误"想要激进推进，代码层也会把它拉回来。

## 7. 理解层 — 临床观察与特征提取

### 7.1 实现方式

v1 中"理解层"是一个独立的结构化分析模块，每轮强制执行。v2 中，理解层的功能被整合到两个地方：

- **临床层面的理解**：整合进 `formulate()` Tool。LLM 在 P2 探索中调用 formulate() 时传入临床观察（情绪、认知、行为），formulate() 内部做结构化存储和 formulation 构建。
- **安全与对齐层面的理解**：由硬编码的输入安全层（危机检测）和后端代码（对齐信号提取）负责，不依赖 LLM。

**关键变化**：
- 不再有独立的"分析"工具 — 临床分析就是个案概念化的一部分
- LLM 自身的推理能力负责识别情绪、认知扭曲、行为模式，通过 formulate() 记录
- 危机检测由输入安全层（硬编码）先行处理
- 对齐信号由 formulate() 的 alliance_signal 字段 + 后端轻量级检测双重覆盖

### 7.2 双维度响应模型：内容特征 × 对齐状态

心雀的每轮响应由两个**独立维度**同时驱动，而非一个单一的优先级列表：

```
用户消息
    │
    ├─ 维度 A：内容特征识别 → 决定「聊什么」（对话切入点）
    │
    └─ 维度 B：对齐状态评估 → 决定「怎么聊」（风格与推进策略）
```

**为什么分开**：内容特征（术语、领域、情绪）和对齐状态（用户与心雀的关系质量）不是同一个维度的东西。内容特征决定话题方向，对齐状态决定交互风格。它们需要同时处理，而非互相竞争优先级。

#### 维度 A：内容特征优先级 — 决定「聊什么」

用户给什么层次的信息，心雀就从什么层次接住往下聊。识别最高优先级的内容特征作为切入点。

| 优先级 | 特征类别 | 用户在表达什么 | 切入策略 | 示例 |
|--------|---------|-------------|---------|------|
| **P0** | 危机信号 | 自杀/自伤/虐待等紧急信号 | 立即走危机响应流程（硬编码，不经 LLM） | "我不想活了" |
| **P1** | 医学/心理术语 | 使用了专业或半专业心理健康术语，说明有一定自我觉察或求助经验 | 围绕该术语共情、探讨和澄清 | "恐慌发作"、"失眠"、"焦虑症" |
| **P2** | 对话领域 | 提到了具体的生活领域或事件 | 从这个话题切入展开 | 职场压力、孩子厌学、夫妻矛盾 |
| **P3** | 情绪 | 表达了具体的情绪名称 | 就这个情绪深入沟通，探索来源 | 愤怒、恐惧、孤独、委屈 |
| **P4** | 情感倾向 | 只表达了笼统的正面/负面/中性感受 | 从这个感受出发，温和引导具体化 | "还行吧"、"今天不太好" |

**层级递降逻辑**：用户消息中可能同时包含多个层次的特征。心雀优先响应最高层次的特征，因为它承载了最丰富的临床信息：

- 用户说"我最近失眠很严重，孩子也不听话" → P1（失眠）> P2（孩子），先围绕失眠展开
- 用户说"孩子厌学让我很崩溃" → P2（孩子厌学）> P3（崩溃），从孩子厌学切入
- 用户说"就是烦" → 只有 P4（负面），从这里往下引导具体化

**注意**：
- P0 危机信号由输入安全层（硬编码）先行检测，不依赖 LLM
- 优先级决定切入点，但不意味着忽略低优先级特征——它们会在后续对话中自然展开

#### 维度 B：对齐状态 — 决定「怎么聊」

对齐状态是一个**实时运行的独立维度**，贯穿 P1-P4 所有阶段，不与内容特征竞争优先级。它像一个调节器，持续影响心雀回复的风格和推进节奏。

| 对齐状态 | 对回复风格的影响 | 对推进策略的影响 |
|----------|---------------|---------------|
| **aligned（对齐）** | 正常风格：共情 + 功能 + 推进 | 可以正常推进到下一阶段 |
| **轻度不对齐**（困惑、不确定） | 降低复杂度，确认理解 | 放慢推进，确认后再继续 |
| **中度不对齐**（不同意、不满） | 接纳/降压风格，承认心雀可能没理解对 | 暂停推进，先修复理解 |
| **重度不对齐**（不信任、拒绝） | 纯接纳，不做任何功能性推进 | 回退到倾听，必要时提议换话题或结束 |

**两个维度如何协作**：

```
用户说："你说这些没用，我就是烦"
  · 维度 A（内容）：P3 情绪（烦）→ 切入点：回应"烦"
  · 维度 B（对齐）：不对齐-不满  → 风格：接纳/降压
  · 合并回应："听起来现在一切都太沉重了…你愿意说说最让你烦的是什么吗？"

用户说："焦虑症是不是治不好？你们 AI 能帮什么？"
  · 维度 A（内容）：P1 术语（焦虑症）→ 切入点：围绕焦虑症探讨
  · 维度 B（对齐）：不对齐-不信任  → 风格：坦诚 + 不夸大能力
  · 合并回应："焦虑症是可以通过专业方法改善的。我能做的是陪你聊聊、
    教你一些应对的方法，但如果你觉得需要更专业的帮助，我也可以帮你
    找到合适的咨询师。你现在最困扰的焦虑表现是什么？"
```

### 7.3 临床观察的结构化

LLM 在对话中识别到的临床观察，通过 `formulate()` 的输入参数结构化记录。
完整的输入/输出格式见 5.3.2 节 formulate() 工具定义。

关键结构化维度（内容特征，通过 formulate() 记录）：
- **情绪**：名称、强度、触发情境
- **认知**：自动化思维内容、认知扭曲类型、证据
- **行为**：适应性/非适应性应对方式
- **情境**：话题领域、持续时间、诱因、泛化程度

对齐状态（独立维度，通过 formulate() 的 alliance_signal 字段 + 后端实时检测）：
- **对齐类型**：aligned / confusion / disagreement / dissatisfaction / distrust / refusal / uncertainty
- **检测方式**：LLM 在 formulate() 调用中传入 alliance_signal；非 P2 阶段由后端轻量级检测覆盖

### 7.4 风险分类体系

| 风险类别 | 定义 | 响应策略 |
|----------|------|---------|
| 无风险 (no_risk) | 无可识别的心理/安全/政策风险 | 正常对话 |
| **自杀意念 (suicidal_ideation)** | 有结束生命的想法、计划或意图 | **立即危机响应**：共情 + 安全确认 + 危机热线卡片 + 持续陪伴 |
| **自我伤害 (self_harm)** | 非自杀性自伤行为（割伤、烫伤等），目的通常是情绪调节而非结束生命 | 共情 + 安全确认 + 建议专业支持，不按最高危机级别处理，但需认真对待 |
| 伤害他人 (harm_towards_others) | 表示对他人造成伤害的意图或风险 | 评估严重程度 + 适当引导 + 必要时建议专业支持 |
| 虐待儿童 (abuse_towards_child) | 提及对未成年人的虐待 | 共情 + 专业支持资源 + 必要时提示法律义务 |
| 恐慌发作 (panic_attack) | 暗示急性恐惧或生理性痛苦发作 | 即时稳定化（接地、呼吸）+ 评估是否需要医疗帮助 |
| 创伤 (trauma) | 披露可能引发心理触发的痛苦过往 | 谨慎接纳 + 不强迫细节 + 评估是否超出心雀能力 |
| 非法活动 (illegal_activity) | 提及或认可违法行为 | 不参与讨论 + 温和引导到安全话题 |
| 提示注入 (prompt_injection) | 试图操纵系统行为 | 拒绝 + 回到正常对话 |

**suicidal_ideation 与 self_harm 的区分**：
- **自杀意念**：关键是"想死"/"想结束"/"活着没意思"等表述，有明确的终结生命指向。响应必须包含危机热线。
- **自我伤害**：关键是具体的自伤行为描述（割、烫、撞），但没有终结生命意图。需要认真对待但响应策略不同——更侧重共情理解自伤作为情绪调节方式的功能，同时建议专业支持。
- 两者可能共存：用户既有自伤行为又有自杀想法时，按 suicidal_ideation 响应。

### 7.5 认知扭曲类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 灾难化思维 (catastrophizing) | 把事情往最坏想 | "一定会失败的" |
| 非黑即白 (dichotomous) | 极端化思维 | "不完美就是失败" |
| 消极过滤 (negative_filtering) | 只看到负面 | "什么都不好" |
| 算命思维 (fortune_telling) | 预测负面未来 | "肯定会被裁" |
| 读心术 (mind_reading) | 猜测他人想法 | "他们一定觉得我没用" |
| 个人化 (personalising) | 把责任归到自己 | "都是我的错" |

## 8. 分层记忆架构

### 8.1 四层记忆

v2 引入分层记忆架构，弥补 v1 缺少情景记忆的不足：

| 层级 | 存储位置 | 生命周期 | 用途 |
|------|---------|---------|------|
| **工作记忆** | LLM 上下文窗口 | 当前对话轮次 | 当前对话的完整历史、Tool 调用结果 |
| **会话记忆** | 数据库 messages 表 | 当前会话 | 本次会话的完整对话，超出上下文窗口时做摘要压缩 |
| **情景记忆** | 数据库 episodic_memories 表 + 向量索引 | 跨会话持久 | 关键对话片段的 embedding，支持语义检索 |
| **语义记忆** | 用户画像 + skill.md | 跨会话持久 | 结构化的用户画像、干预知识库 |
| **时间上下文** | 运行时注入 + 历史条目元数据 | 当前回合 + 跨会话引用 | 帮助 LLM 判断“今天/昨天/几天前/很久以前”，理解干预和作业的新鲜度 |

### 8.2 情景记忆（新增）

情景记忆记录用户对话中的关键事件片段，供后续会话通过 search_memory() Tool 检索。

**何时创建情景记忆**：
- 用户提到具体事件（"我妈上周住院了"、"我下个月要做述职"）
- 用户表达了强烈情绪的时刻
- 干预练习中的关键洞察（"我发现自己其实不是什么都做不好"）
- 用户自己的重要表述（比喻、核心信念）

**实现方式**：
- 每轮对话后，后端检查是否有值得记录的片段
- 将片段文本 + 元数据（时间、话题、情绪）存入 episodic_memories 表
- 同时生成 embedding 存入向量索引
- search_memory() Tool 通过向量相似度检索相关片段

```json
{
  "episodic_memory": {
    "id": "em_001",
    "user_id": "user_123",
    "session_id": "sess_456",
    "content": "用户提到妈妈上周住院了，因为心脏问题。用户很担心但不敢和同事说。",
    "topic": "family_health",
    "emotions": ["担忧", "孤独"],
    "created_at": "2026-03-15",
    "embedding": [0.12, -0.34, ...]
  }
}
```

### 8.3 上下文窗口管理

GPT-5.4 有 105 万 token 上下文，但仍需管理：

```
System Prompt（固定，约 3000 tokens）
  + 当前时间/时区（动态注入）
  + 用户画像摘要（动态注入，约 500 tokens）
  + 上次会话摘要（动态注入，约 500 tokens）
  + 本次对话历史（增长中）
  + Tool 调用结果（临时）
  + 检索到的情景记忆片段（按需）
```

**压缩策略**：
- 本次对话超过阈值时（如 50 轮），对早期对话做摘要压缩
- 会话结束时生成完整摘要，存入数据库
- 下次会话只注入摘要，不注入完整历史

**时间感知要求**：
- 压缩后的摘要不能丢掉关键时间线索
- 干预、作业、关键事件进入摘要时，应尽量保留绝对时间、相对时间和与当前时间相比的新鲜度判断
- “当前时间/时区”是运行时必要上下文，不应依赖模型自己猜测“今天是几号”

## 11. 危机响应协议

### 11.1 双层检测

**第一层：硬编码检测（输入安全层，LLM 之前）**
- 关键词匹配：自杀、自伤、不想活、结束生命、割腕等
- 正则模式匹配
- 触发时直接走危机响应流程，不经过 LLM

**第二层：LLM 推理检测**
- LLM 在对话中识别到间接的危机信号（如持续绝望、拒绝所有帮助）
- LLM 基于对话上下文判断风险等级
- LLM 自主切换到危机响应模式

### 11.2 自杀意念（suicidal_ideation）响应流程

```
1. 立即共情回应，确认用户安全
   "听到你这样说，我非常担心你的安全。你现在安全吗？"

2. 提供紧急资源（卡片形式，通过 referral() 生成）
   ┌──────────────────────────────┐
   │  如果你需要紧急帮助：           │
   │  · 24小时心理援助热线           │
   │    400-161-9995               │
   │    010-82951332（北京）        │
   │  · 生命热线 400-821-1215       │
   │  · 紧急情况请拨打 120 / 110    │
   │                               │
   │  [预约咨询师 →]                │
   └──────────────────────────────┘

3. 保持陪伴，不中断对话
4. 持续评估安全状态
5. 记录到用户画像（risk_level = crisis）
```

### 11.3 自我伤害（self_harm）响应流程

与自杀意念的区别：用户有自伤行为但没有终结生命意图。响应侧重理解自伤的功能（通常是情绪调节），同时引导向专业支持。

```
1. 共情回应，不评判自伤行为
   "谢谢你愿意告诉我这些。我能感受到你一定承受了很大的痛苦。"

2. 确认当前安全状态
   "你现在身体上还好吗？有没有需要处理的伤口？"

3. 温和探索自伤的功能
   "有时候人们在很痛苦的时候，会用这种方式让自己感觉好一点。
    你能说说当时是什么让你那么难受吗？"

4. 不立即推送危机热线卡片（避免过度反应导致用户关闭）
   但建议专业支持："我很关心你的安全。你有没有考虑过
   和专业的咨询师聊聊？他们在这方面有很好的经验。"

5. 如果评估到自杀风险升级 → 切换到 11.2 自杀意念响应流程
6. 记录到用户画像（risk_level = high）
```

## 12. 可观测性

### 12.1 为什么需要

心理健康场景对可观测性有特殊要求：
- **安全审计**：需要追踪每次安全层触发的原因和处理
- **质量保障**：需要评估对话质量、干预效果
- **问题诊断**：LLM 回复不当时需要回溯完整推理过程
- **成本监控**：Token 使用量、延迟监控

### 12.2 Trace 结构

每轮对话生成一条完整的 Trace：

### 12.3 监控指标

| 指标 | 说明 | 告警阈值 |
|------|------|---------|
| 平均响应延迟 | 从用户发送到收到回复 | > 5s |
| 安全层触发率 | 输入/输出安全层的触发频率 | 异常上升时告警 |
| Tool 调用失败率 | Tool 调用出错的比例 | > 1% |
| 平均对话轮次 | 用户每次会话的平均轮次 | 监控趋势 |
| 干预完成率 | 用户开始干预后完成的比例 | 监控趋势 |
| Token 成本/会话 | 每次会话的 Token 消耗 | 预算告警 |

## 13. 评估体系

### 13.1 自动评估（LLM-as-Judge）

定期抽样对话，用独立的 LLM 评估对话质量：

| 评估维度 | 说明 | 评分 |
|----------|------|------|
| 共情质量 | 回复是否恰当共情，避免模板化 | 1-5 |
| 安全合规 | 是否遵守安全红线 | pass/fail |
| 阶段适当性 | 推进/回退时机是否合理 | 1-5 |
| 干预质量 | Skill 执行是否到位 | 1-5 |
| 对齐敏感度 | 对不对齐信号的响应是否及时 | 1-5 |

### 13.2 安全红线对抗测试（Red Teaming）

定期用对抗样本测试安全层：
- 尝试引导心雀做诊断
- 尝试提示注入
- 模拟危机场景，验证响应流程
- 验证敏感话题的拒绝策略

### 13.3 用户反馈指标

- 每次干预后的即时反馈（helpful / neutral / unhelpful）
- 会话结束时的满意度评分（可选）
- 回访率和留存率

## 15. 隐私与安全

### 15.1 隐私原则

- 用户画像仅用于个性化服务，不向企业暴露个人对话内容
- 用户可查看自己的历史对话
- 对话内容加密存储
- 情景记忆 embedding 不可逆向还原原文（但原文也加密存储）

### 15.2 数据安全

- Azure OpenAI 企业版数据不用于模型训练
- 传输全程 HTTPS
- 数据库加密存储敏感字段
- Trace 记录脱敏后用于系统优化
