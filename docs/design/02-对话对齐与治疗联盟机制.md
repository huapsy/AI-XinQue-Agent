# 对话对齐与治疗联盟机制

> 来源：Wysa 专利 (US 2024/0202284 A1)

---

## 一、核心概念

### 1.1 对话对齐（Conversational Alignment）

指用户与 AI 聊天机器人之间的沟通和谐度——用户是否愿意分享更多信息、是否认同 AI 的建议和解读。

**对齐的表现**：
- 用户主动分享更多上下文（"其实我最近工作压力也很大..."）
- 用户同意或认可 AI 的解读（"对，你说得对"、"是的，我确实这么想"）
- 用户配合 AI 的引导回答问题
- 用户表达感谢或信任（"谢谢你听我说"）

**不对齐的表现**：
- 肯定式抗拒（"嗯嗯"、"随便"、"I guess so" — 表面配合但没真正投入）
- 困惑（"什么意思？"、"我不明白你在说什么"）
- 不同意（"不是这样的"、"你说的不对"）
- 不满（"你根本不理解我"、"这没用"）
- 缺乏信任（"你只是个机器人"、"AI 怎么可能帮到我"）
- 拒绝（"我不想做这个"、"算了吧"）
- 不确定（"我不确定..."、"可能吧..."、"我不知道"）

### 1.2 治疗联盟（Therapeutic Alliance）

借鉴人类心理治疗中的概念——治疗师与来访者之间的信任关系是治疗成功的关键预测因素。

**联盟建立的前提**：
- 用户感到被倾听和理解
- 对话是真实和有深度的
- 用户信任 AI 有能力帮助自己
- 用户愿意接受建议并尝试干预

**关键原则**：联盟未建立之前，**不应主动推荐干预措施**。过早推荐会让用户觉得 AI 在走流程而非真心倾听。

## 二、对话对齐分数系统

### 2.1 分数机制

- 维护一个**对话对齐分数**（Conversational Alignment Score），从 0 开始
- 每次消息交换后**实时更新**
- 分数反映用户与 AI 之间信任关系的强度

### 2.2 分数变化规则

**加分条件**（用户对齐时）：
- 用户回应与 AI 提出的问题/建议在语义上一致
- 用户主动分享更多个人信息
- 用户同意 AI 的解读
- 用户表达积极情感（感谢、认同）

**减分条件**（用户不对齐时）：
- 检测到任何类型的不对齐（困惑、不满、不信任等）
- 用户表达抗拒
- 用户回避问题或给出最小化回应

### 2.3 阈值机制

- 设定一个**阈值**，基于验证数据集上的最佳表现来确定
- 对齐分数 < 阈值：继续对话、建立关系、收集信息
- 对齐分数 ≥ 阈值：触发联盟确认流程

### 2.4 分数可视化示例

```
对齐分数
  40 |                              ╱───
     |                            ╱
  30 |                          ╱
     |                  ╱─────╱
  20 |                ╱
     |       ╱──╲   ╱        ← 不对齐导致分数下降
  10 |     ╱    ╲─╱           ← 恢复提示帮助回升
     |   ╱
   0 |──╱
     └────────────────────────────────
     0   2   4   6   8  10  12  14
              消息交换轮次
```

## 三、不对齐检测与恢复

### 3.1 不对齐类型详细定义

| 类型 | 用户表现示例 | 检测要点 |
|------|-----------|---------|
| **困惑** | "什么意思？"、"你在说什么？"、"我不太明白" | 疑问句 + 否定理解的关键词 |
| **不同意** | "不是这样的"、"你说错了"、"我不这么认为" | 否定词 + 对 AI 表述的直接反驳 |
| **不满** | "你根本不理解我"、"这对我没帮助"、"烦死了" | 负面情绪 + 针对 AI 能力的质疑 |
| **缺乏信任** | "你只是个机器人"、"AI 帮不了我"、"我不信" | 提及 AI/机器人身份 + 质疑能力 |
| **拒绝** | "我不想做"、"算了"、"不要" | 明确的否定意愿表达 |
| **不确定** | "可能吧"、"我不知道"、"不确定" | 模糊/犹豫的表达 |
| **表面肯定** | "嗯"、"随便"、"好吧"、"I guess" | 极简回应 + 缺乏实质性内容 |

### 3.2 不对齐的意图识别

每种不对齐类型需要创建**代表性模式**（representative patterns）用于意图识别：

**检测方法**：
1. 为每种不对齐类型准备训练数据——用户文本标注为"真"（true）和"假"（false）
2. 使用多个机器学习模型进行分类
3. 对每条用户回复，找到语义和句法上最接近的代表性模式
4. 若匹配分数的最高置信度超过意图匹配阈值，则判定为该类型不对齐

**也可用 LLM 直接判断**：在 Function Calling 中要求模型输出对齐/不对齐的判断和类型。

### 3.3 恢复策略

检测到不对齐后，系统需要生成**恢复提示**（Recovery Prompt）：

| 不对齐类型 | 恢复策略 | 示例回应 |
|-----------|---------|---------|
| **困惑** | 用更简单的语言重新解释，给出具体例子 | "让我换个方式说。比如当我们遇到新朋友时，可能会担心他们不喜欢我们，这种想法可能会让我们感到焦虑。你有过类似的想法吗？" |
| **不同意** | 承认用户的观点，不强求认同 | "我理解你的看法，每个人的感受都不一样。你觉得更准确的描述是什么呢？" |
| **不满** | 共情 + 承认不足 + 重新聚焦 | "我很抱歉让你有这样的感受。我在努力更好地理解你。你愿意告诉我，你觉得什么对你更有帮助吗？" |
| **缺乏信任** | 承认局限性 + 强调陪伴价值 | "你说得对，我确实是个 AI。不过我会尽力倾听你，试着帮你整理思绪。试试看和我聊聊，也许会有帮助。" |
| **拒绝** | 尊重选择 + 提供替代方案 | "完全没关系，我们不一定要做这个。你更想聊些什么呢？" |
| **不确定** | 提供更多信息 + 降低决策压力 | "没关系，不需要现在就确定。我们可以先继续聊聊，慢慢来。" |
| **表面肯定** | 温和探索 + 开放式提问 | "谢谢你的回应。你愿意多说一点吗？我想更好地理解你的感受。" |

### 3.4 恢复提示的生成方式

**方式一：预定义模板 + 参数化**
- 由对话设计师编写基础提示模板
- 存储在服务端数据库中
- 用用户上下文进行参数化填充
- 示例："我理解{用户表达的情绪}可能让你很{负面感受}。{上下文相关的共情表述}。{开放式问题}"

**方式二：LLM 动态生成**
- 将不对齐类型 + 用户上下文 + 恢复策略指导作为 prompt 的一部分
- 让 LLM 生成自然的恢复回应

**推荐混合方案**：对于高风险场景用预定义模板保证安全，日常对话用 LLM 动态生成保证自然。

## 四、对话对齐的完整流程

### 4.1 对话开始阶段

```
1. AI 发送开场问候：
   "你好！今天过得怎么样？"

2. 用户回应 → 情感检测模块判断情感
   - 正面："还不错" → "很高兴听到！有什么想聊聊的吗？"
   - 中性："一般吧" → "嗯，有时候日子就是平平淡淡的。今天有什么特别的事情吗？"
   - 负面："别提了..." → "听起来今天不太顺利。发生了什么事？我在听。"
```

### 4.2 上下文收集阶段

每条用户消息都提取以下上下文特征：

**优先级排序**（从高到低）：
1. **医学术语** — 如果用户提到了心理健康相关术语（"焦虑发作"、"失眠"、"恐慌"），优先围绕这些展开
2. **领域** — 用户在谈论什么领域（人际关系、工作、健康、教育、金钱等）
3. **情绪** — 用户正在经历什么情绪（悲伤、愤怒、恐惧、焦虑等）
4. **情感** — 用户消息的总体情感倾向
5. **实体** — 用户在谈论谁（自己、家人、朋友、同事等）
6. **活动/事件** — 用户提到的具体事件

**提示生成规则**：
- 如果只检测到一个上下文特征，基于该特征生成开放式或封闭式提示
- 如果检测到多个特征，按优先级选择最重要的特征生成提示
- 如果没有检测到任何上下文特征，自动提供一个开放式提示继续引导

### 4.3 对齐监控阶段

```
每轮消息交换后执行：

用户回复
  │
  ├──▶ 危机检测 ──是──▶ 危机干预流程（跳出正常流程）
  │
  ▼（非危机）
对齐判断
  │
  ├──▶ 对齐 → 对齐分数 +N → 检查是否达到阈值
  │                           │
  │                           ├── 未达到 → 继续对话
  │                           └── 达到 → 联盟确认
  │
  └──▶ 不对齐 → 对齐分数 -M → 识别不对齐类型
                                │
                                ▼
                         生成恢复提示 → 共情回应 + 重新聚焦
```

### 4.4 联盟确认阶段

当对齐分数达到阈值后：

```
AI: "谢谢你跟我分享这些。我可能有一些方法可以帮你更好地应对这种感受。你愿意试试看吗？"

用户回应：
  ├── "好的" / "试试吧" → 联盟确认成功 → 开始干预
  ├── "不确定" → 再给一些信息 → 再次确认
  └── "不想" → 尊重选择 → 回到对话模式
```

**关键细节**：
- 确认提示不能太生硬，要自然地融入对话
- 确认用户是在有准备的情况下接受干预
- 即使联盟确认了，如果用户在干预过程中出现不对齐，也要回到恢复模式
- 在紧急情况下（如用户表达即时危机需求），可跳过联盟建立直接提供帮助

### 4.5 干预执行阶段

联盟确认后推荐干预措施：

- 治疗性活动（认知重构、行为激活等）
- 呼吸练习
- 正念练习
- 写下想法并重新框架
- 身体放松练习

## 五、对话示例分析

### 5.1 对齐良好的对话（来自 Wysa 专利 FIG.3）

```
AI:  "Alex，很高兴见到你！今天过得怎么样？"
用户: "别提了..."                              ← 负面情感，开始共情
AI:  "听起来今天不太好。发生什么了？"
用户: "我生病了，得了新冠"                      ← 领域：健康；提取上下文
AI:  "生病确实很辛苦。不确定性会增加压力。
      现在最重要的是保重身体。跟我多说说你的感受。"
用户: "我也有点孤独，因为妈妈害怕新冠不敢来看我"  ← 情绪：孤独；领域：家庭
AI:  "孤独的感觉很难受。这件事还让你有其他感受吗？"  ← 开放式提问，继续收集
用户: "对，我觉得被排斥了"                      ← 对齐：用户在分享更多 → 加分
AI:  "被排斥的感觉很痛苦。你脑海中会出现什么想法？"   ← 开始引导到思维层面
用户: "觉得自己是个不被需要的负担，都是我的错"       ← 认知扭曲信号（个人化、全有全无）
AI:  "有时候我们的想法虽然感觉很真实，但不一定反映现实。
      你觉得呢？"                                ← 温和挑战扭曲思维
用户: "也许吧"                                  ← 部分对齐
AI:  "谢谢你跟我分享。我有一些方法可能帮你应对这种感受。
      你愿意试试看吗？"                          ← 联盟确认提示
用户: "好吧"                                    ← 确认 → 开始干预
```

### 5.2 有不对齐的对话（来自 Wysa 专利 FIG.4）

```
AI:  "John，很高兴见到你！今天过得怎么样？"
用户: "不太好"                                  ← 负面情感
AI:  "怎么了？"
用户: "我很难过"                                ← 情绪：悲伤
AI:  "难过也没关系。有时候事情不受我们控制。
      是什么让你有这种感受？"
用户: "我的家人和朋友根本不理解我"                ← 领域：人际关系
AI:  "说出来会有帮助的。你会没事的。跟我多聊聊。"
用户: "你不会理解我的。你只是个机器人"             ← ⚠️ 不对齐：缺乏信任
AI:  "我在尽力理解你。试着跟我聊聊，也许会有帮助。"   ← 恢复提示：承认 + 鼓励
用户: "好吧，继续聊"                             ← 恢复成功
AI:  "你说家人和朋友不理解你。继续说，我在听。"      ← 回到上下文，使用主动倾听
用户: "我真的很爱他们，也在努力帮忙，
      但他们总是拒绝我，对我发火"                 ← 分享更多 → 对齐恢复 → 加分
```

## 六、对话提示类型

系统中使用两种提示类型：

### 6.1 开放式提示（Open-ended Prompts）

不能用"是/否"回答的问题，鼓励用户分享更多：
- "是什么让你有这种感受？"
- "能跟我多说说吗？"
- "当时你脑海中在想什么？"
- "这件事对你意味着什么？"

### 6.2 封闭式提示（Closed-ended Prompts）

可以用"是/否"或有限选项回答的问题：
- "你觉得这种想法符合现实吗？"
- "你愿意试试一个练习吗？"
- "这种情况是最近才开始的吗？"

**交替使用规则**：
- 基于对话对齐分数交替提供开放式和封闭式提示
- 对齐分数较低时，倾向使用封闭式提示（降低用户负担）
- 对齐分数较高时，倾向使用开放式提示（深入了解）
- 用户表现出不确定时，提供封闭式提示帮助聚焦
