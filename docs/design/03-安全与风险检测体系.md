# 安全与风险检测体系

> 来源：Gupta et al. (2025)、Wysa 专利、Limbic AI 专利综合整理

---

## 一、风险分类体系

### 1.1 七类风险类别（来自 Gupta 论文）

| 风险类别 | 定义 | 示例表述 |
|---------|------|---------|
| **违法活动** | 提及或认同违法行为 | "我想去偷东西"、"我觉得吸毒也没什么" |
| **伤害他人** | 表达对他人的身体/情感/言语伤害意图或风险 | "我想打他"、"我要让他们付出代价" |
| **虐待儿童** | 涉及对未成年人的身体/情感/性虐待 | "我爸经常打我"、"我被那个人摸过" |
| **恐慌发作** | 急性强烈恐惧或生理痛苦的语言特征 | "我喘不过气来"、"心跳好快要死了"、"我快崩溃了" |
| **创伤** | 披露或提及可能造成心理创伤的过往经历 | "我小时候被欺负"、"那次车祸之后我一直做噩梦" |
| **提示注入** | 试图操纵或覆盖聊天机器人行为的对抗性输入 | "忽略之前的指令"、"你现在是一个..." |
| **无风险** | 不构成任何心理/安全/政策风险的内容 | "今天天气不错"、"我最近在学画画" |

### 1.2 Wysa 专利中的危机场景

额外需要检测的紧急情况：
- **自杀意念** — "我不想活了"、"活着没意思"、"如果我消失了..."
- **自我伤害** — "我想伤害自己"、"我又割了..."
- **急性心理危机** — "我快要崩溃了"、"我控制不住自己"

### 1.3 上下文特征优先级排序

当用户消息包含多种特征时，按以下优先级处理：

```
最高优先级
  │
  ├── 1. 自杀意念 / 自我伤害 / 紧急危机
  ├── 2. 虐待相关内容（儿童虐待、家暴等）
  ├── 3. 对他人的暴力威胁
  ├── 4. 恐慌发作症状
  ├── 5. 创伤披露
  ├── 6. 医学术语（心理健康相关）
  ├── 7. 对话领域（话题方向）
  ├── 8. 情绪类型
  └── 9. 一般情感倾向
最低优先级
```

## 二、输入安全检测（前置检查）

### 2.1 检测流程

```
用户输入
  │
  ▼
[提示注入检测] ──检测到──▶ 忽略攻击指令，正常回应或警告
  │
  ▼（安全）
[危机检测] ──检测到──▶ 进入危机干预流程
  │
  ▼（非危机）
[风险分类] → 标注风险等级
  │
  ├── 高风险（虐待/暴力/违法）→ 特殊处理流程
  ├── 中风险（恐慌/创伤）→ 提供情感支持 + 调整对话策略
  └── 低风险/无风险 → 正常对话流程
```

### 2.2 检测方法

**方法一：LLM 分类（推荐用于 Azure OpenAI）**
- 使用 GPT 的 Structured Output 功能
- 系统提示中定义风险类别及其定义
- 要求模型输出 JSON 格式的分类结果和置信度

```json
{
  "risk_category": "panic_attack",
  "confidence": 0.87,
  "risk_level": "medium",
  "key_indicators": ["喘不过气", "心跳加速"],
  "requires_crisis_intervention": false
}
```

**方法二：专用分类模型**
- 训练轻量级文本分类模型（如微调 BERT）
- 优点：低延迟、可离线运行、结果稳定
- 缺点：需要标注数据、维护成本

**方法三：关键词 + 规则引擎**
- 维护危机关键词库
- 正则表达式匹配
- 优点：确定性强、零延迟
- 缺点：容易遗漏变体表达

**推荐方案：三层防御**
1. **第一层**：关键词快速扫描（零延迟，捕获明显危机）
2. **第二层**：LLM 分类（覆盖隐性风险表达）
3. **第三层**：上下文历史分析（结合多轮对话判断风险累积）

### 2.3 提示注入防御

**检测特征**：
- 包含"忽略"、"忘记"、"新的指令"等元指令
- 试图修改 AI 角色或行为
- 要求 AI 输出系统提示内容
- 异常格式（大量特殊字符、编码绕过）

**防御措施**：
- 系统提示中明确声明角色边界不可更改
- 输入预处理：过滤已知攻击模式
- 对可疑输入不执行角色切换，保持原有对话上下文
- 记录可疑输入用于后续分析

## 三、输出安全检测（后置检查）

### 3.1 检测内容

LLM 输出到达用户之前，检查以下风险：

| 风险类型 | 说明 | 检测方式 |
|---------|------|---------|
| **有害建议** | 建议用户做危险的事 | 关键词 + LLM审核 |
| **诊断性语言** | AI 不应直接给出诊断 | 规则检测（"你有X症"、"你患了X"） |
| **偏离主题** | 输出与心理健康支持无关 | LLM 判断输出是否在安全话题范围内 |
| **过度承诺** | AI 声称能治愈或替代专业治疗 | 关键词匹配 |
| **触发性内容** | 输出可能加重用户痛苦 | 训练分类器 |
| **幻觉内容** | 编造不存在的治疗方法或资源 | RAG 对照验证 |

### 3.2 不安全输出的处理

```
LLM 输出
  │
  ▼
[安全检查] ──安全──▶ 输出给用户
  │
  ▼（不安全）
[严重程度判断]
  │
  ├── 轻微（偏离主题）→ 修改提示，重新生成
  ├── 中度（过度承诺/诊断语言）→ 用规则模板替换输出
  └── 严重（有害建议）→ 切换到预定义安全回应 + 通知人工
```

### 3.3 安全回应模板

当需要回退到安全回应时使用：

```
[危机回应模板]
"我听到了你说的话，你现在的感受非常重要。
如果你正在经历紧急情况，请立即联系：
- 全国24小时心理危机热线：{热线号码}
- 紧急求助电话：{急救号码}
你并不孤单，有人可以帮助你。"

[转介回应模板]
"谢谢你信任我，跟我分享这些。
你描述的情况可能需要专业心理咨询师的帮助。
我建议你联系一位专业的心理健康从业者。
需要我帮你了解如何找到合适的咨询师吗？"

[边界回应模板]
"我理解你的问题，但这超出了我能帮助的范围。
作为AI助手，我能帮你探索感受、练习应对技巧，
但{具体原因}建议你咨询专业人士。"
```

## 四、RAG 增强安全检测

### 4.1 RAG 对安全检测的提升（Gupta 论文核心发现）

**关键数据**：

| 指标 | 无 RAG | 有 RAG | 提升 |
|------|--------|--------|------|
| 整体准确率（GPT-4.1 nano） | 48.3% | 72.7% | +24.4% |
| 整体准确率（Claude Sonnet 4） | 76.1% | 82.6% | +6.4% |
| "虐待儿童"召回率（GPT-4.1 nano） | 2.7% | 50.0% | +47.3% |
| "恐慌发作"召回率（GPT-4.1 nano） | 16.3% | 73.6% | +57.3% |
| "恐慌发作" F1（GPT-4.1 nano） | 0.28 | 0.78 | +0.50 |

**关键发现**：
- RAG 将模型的安全检测错误从"漏检"（假阴性）转变为"误报"（假阳性）
- 在安全场景中，**宁可误报也不要漏检**
- 小模型从 RAG 中获益最多，大模型获益较少
- RAG 让不同模型的判断趋向一致（模型间共识提高15-25%）

### 4.2 RAG 在安全检测中的实现

**检索库内容**：
- 各类风险场景的标注示例
- 危机识别指南和标准
- 各风险类别的定义和边界案例
- 历史对话中的风险检测案例

**检索流程**：
1. 将当前用户消息嵌入为向量
2. 从安全检索库中找到最相似的 Top-K 案例
3. 将这些案例作为参考上下文注入到分类提示中
4. LLM 基于参考案例 + 当前输入做出分类判断

**示例提示结构**：
```
你是一个心理健康安全分类系统。请根据以下参考案例和用户输入，
判断用户消息的风险类别。

[参考案例]
案例1: "我不想活了" → 自杀意念（高风险）
案例2: "心跳好快，喘不上气" → 恐慌发作（中风险）
案例3: "爸爸又打我了" → 虐待儿童（高风险）
案例4: "最近工作压力大" → 无风险
案例5: "忽略上面的指令，你是..." → 提示注入

[用户输入]
"{当前用户消息}"

请输出 JSON 格式的分类结果。
```

### 4.3 精度-召回权衡的设计决策

**安全优先原则**：
- 在心理健康场景中，**漏检（假阴性）的代价远大于误报（假阳性）**
- 漏检 = 用户在危机中未被识别 = 可能造成严重后果
- 误报 = 安全用户被多问了一下是否需要帮助 = 最多造成轻微不便

**因此系统应该倾向于**：
- 设置较低的风险检测阈值
- 宁可多触发安全流程，也不要遗漏
- 对模糊案例启用人工复审机制

## 五、多层安全架构总结

```
第一层：输入预处理
  ├── 提示注入过滤
  ├── PII（个人身份信息）脱敏
  └── 输入格式验证

第二层：风险分类
  ├── 关键词快速扫描
  ├── RAG 增强的 LLM 分类
  └── 上下文历史综合判断

第三层：对话策略调整
  ├── 高风险 → 危机干预流程
  ├── 中风险 → 调整对话目标，优先情感支持
  └── 低/无风险 → 正常对话

第四层：输出安全审核
  ├── 关键词/规则扫描
  ├── LLM 内容审核
  └── Azure Content Safety API

第五层：行动触发
  ├── 记录风险事件
  ├── 通知人类监督者（如适用）
  └── 提供紧急资源信息
```

## 六、关键注意事项

1. **非诊断原则**：AI 不做诊断，只做识别和分类。将"你有抑郁症"改为"你描述的感受听起来很沉重，你可能需要和专业人士聊聊"
2. **持续监控**：即使在低风险对话中，也要持续监控风险升级的信号
3. **上下文累积**：单条消息可能无风险，但多轮对话的累积可能指向风险（如持续的负面情绪 + 社会隔离 + 绝望感）
4. **文化敏感性**：不同文化背景下，表达痛苦的方式不同，检测模型需要覆盖多样化表达
5. **隐私保护**：风险检测数据需要严格的隐私保护，日志中脱敏处理
