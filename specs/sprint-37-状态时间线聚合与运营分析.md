# Sprint 37 - 状态时间线聚合与运营分析

## 背景

当前心雀已经具备：

- `SessionState`
- `SessionStateHistory`
- `phase_timeline`
- trace 查询接口
- 最小版状态读取接口

但这些能力仍偏“技术可读”，还不够“运营可用”。当前可以查看原始状态与原始 timeline，却还不能稳定回答这些高频问题：

- 这次会话的主要阶段是如何演进的？
- 近几轮为什么从探索切到干预，或者从干预回退到倾听？
- 当前主题是怎么变化的？
- 最近有哪些状态变化是“重要变化”，哪些只是噪声？

在 `plan-v2.1` 已进入生产补强阶段的前提下，这一层属于优先该补的消费层能力。

## 目标

本轮 sprint 的目标，是把当前“可读取的状态/时间线原始数据”升级成“可消费的聚合分析视图”，为运营、评估、回访分析和后续主题连续性治理提供稳定基线。

完成后应达到：

- 能从 API 直接读取聚合后的状态演进与 phase timeline
- 能识别会话中的关键状态切换和重要变化
- 能按时间范围或分页稳定拉取，不依赖前端自己遍历全量 trace

## 范围

### 1. 状态历史读取接口产品化

补齐当前 `state-history` 的消费能力，包括但不限于：

- 分页
- 基于 change reason 的过滤
- 按版本或时间窗口继续拉取
- 最小 continuation metadata

目标不是暴露更多原始 JSON，而是让消费方可以稳定做“继续拉取”和“聚焦关键变化”。

### 2. Timeline 聚合读取

在当前 phase timeline 基础上，补一个会话级聚合视图，至少能表达：

- 每轮出现过哪些 phase
- 关键 phase 切换点
- 关键工具调用与状态变化的关系
- 哪些轮次属于“推进”，哪些轮次属于“回退/修复”

### 3. 运营分析视图最小版

新增面向运营和评估的最小分析载荷，至少支持：

- 当前会话主线摘要
- 最近关键状态变化列表
- phase 演进概览
- 可用于后续 dashboard 的稳定字段

### 4. 测试与文档同步

补齐：

- API 测试
- timeline / state 聚合序列化测试
- 对实现状态文档的同步更新

## 非目标

- 本轮不重写 `SessionState` 数据模型
- 不补完整前端运营后台页面
- 不直接做主题重排策略
- 不引入外部 BI 系统

## 预期结果

如果本轮完成，心雀将从“状态能存能读”推进到“状态能被稳定消费和分析”，为下一轮的主题连续性重排和运营指标建设打下基础。
