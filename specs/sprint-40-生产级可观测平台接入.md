# Sprint 40 - 生产级可观测平台接入

## 背景

当前心雀已经有：

- trace 主链路
- OTel 兼容 exporter
- 匿名统计最小页

但这仍然不等于生产级可观测平台。当前缺口主要体现在：

- 缺少完整 collector / sink / dashboard 路径
- 关键运行指标仍偏“能导出”，不是“可稳定查询和告警”
- 对检索、tool runtime、Responses 调用的观察仍偏离线与手工

随着系统进入生产补强阶段，观测平台需要从“开发辅助”升级为“运行基础设施”。

## 目标

本轮 sprint 的目标，是把心雀已有的 trace / OTel 兼容能力接入更完整的生产级观测链路。

完成后应达到：

- 关键运行事件可进入统一 collector / dashboard
- 主要请求、tool、检索、错误路径有统一观测字段
- 有最小查询面板或运行说明，能支持排障和日常巡检

## 范围

### 1. OTel 链路接入

补齐从应用到 collector 的完整链路，明确：

- exporter 配置
- resource / span / event 字段
- 不同环境下的启停策略

### 2. 关键运行指标视图

至少补齐以下观察对象：

- chat 请求延迟
- Responses API 调用量与失败率
- tool 调用成功率与失败率
- 检索耗时与命中情况
- 安全守门触发情况

### 3. Dashboard / Runbook 最小版

提供最小可用的：

- 仪表板字段设计
- 关键图表或查询说明
- 故障排查入口

### 4. 观测脱敏与边界

任何新增观测链路都必须延续当前隐私约束，避免把敏感原文直接推入外部平台。

## 非目标

- 本轮不做完整 SRE 体系
- 不做企业级告警矩阵全覆盖
- 不重写匿名统计页

## 预期结果

如果本轮完成，心雀将具备“生产环境可观察”的基本形态，而不再只是“本地可调试”。
