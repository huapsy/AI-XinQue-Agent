# Sprint 29 - 全面审查收口与架构重构

## 背景

截至 `Sprint 28`，心雀的主线能力已经基本落地：单一 LLM + Tool Use、长会话状态、时间感知上下文、时间新鲜度路由、管理端匿名统计、会话状态读取接口等都已经具备。

但近期在实际运行中暴露出一批“不是单点 bug，而是结构性问题”的缺口：

1. **数据库 schema 漂移**  
   当前模型已经包含 `clinical_profile`、`SessionState`、`SessionStateHistory` 等字段/表，但 Alembic revision 没有完整覆盖，旧 SQLite 数据库会在真实运行中报 `no such column`。

2. **Tool 协议源头不统一**  
   当前 Tool 定义源头仍保留 Chat Completions 风格，Responses API 侧依赖运行时转换层适配。短期可用，但长期会造成协议认知分裂、测试负担增大。

3. **配置仍是本地开发写法**  
   前端 API 地址、后端 CORS 等仍硬编码为 localhost，离产品化和部署环境切换还有明显距离。

4. **前端产品壳层仍偏 MVP**  
   当前前端核心可用，但仍主要是聊天页 + `#admin`，与 `plan-v2` 中更完整的产品壳层（登录、独立历史页、更清晰导航）有差距。

5. **运行时代码存在“先跑通后适配”的收口债务**  
   包括开发期 schema 兼容、Responses tool schema 适配、状态层与迁移边界、读写路径的职责收口等。

因此需要一个不再新增功能、而是系统性“收口、统一、去临时层”的重构 sprint。

## 目标

通过一次全面收口重构，把当前项目从“高完成度 MVP / 内测版”推进到“更稳定、可维护、接近产品态”的代码结构。

本 sprint 的目标不是继续堆功能，而是：

1. 消除已经暴露出来的结构性运行风险
2. 统一协议源头与迁移边界
3. 把开发期兼容层尽量下沉为可解释、可替换的过渡方案
4. 为后续生产化和产品化迭代建立更干净的基线

## 本 Sprint 范围

### 1. 数据库迁移与 schema 收口

目标：

- 为当前已在模型中使用的字段/表补齐 Alembic revision
- 确保新环境可以通过迁移而不是依赖 `create_all()` + 隐式修补启动
- 开发期兼容修复保留为过渡措施，但不能继续成为正式 schema 管理主路径

重点对象：

- `user_profiles.clinical_profile`
- `session_states`
- `session_state_history`
- 其他已进入代码主链路但未被迁移文件覆盖的字段/表

### 2. Tool schema 源头统一

目标：

- 把 Tool 定义统一成 Responses API 的原生格式
- 消除“源定义一种格式、运行时再转换成另一种格式”的长期维护负担
- 保留必要的兼容测试，但不继续依赖临时转换层作为正式长期架构

重点对象：

- 所有 `TOOL_DEFINITION`
- `responses_helpers.py` 中的协议适配职责
- 与 tool schema 相关的测试

### 3. 配置环境化

目标：

- 去掉前端/后端中明显的 localhost 硬编码
- 通过环境变量或集中配置管理 API 地址、CORS 白名单等
- 保持本地开发默认值可用，但不再把开发地址写死在业务代码里

重点对象：

- 前端 API base URL
- 后端 CORS origin 配置
- 相关文档和启动说明

### 4. 前端产品壳层收口

目标：

- 在不大幅重做 UI 的前提下，把当前前端从“聊天单页 + hash admin”提升为更清晰的应用结构
- 明确聊天、历史、管理页的入口边界
- 为后续登录/身份层接入保留结构位置

本 sprint 不要求做真正企业登录，但要求前端结构不再长期卡在 `#admin` 这种 MVP 路由写法。

### 5. 运行时职责下沉与清理

目标：

- 继续缩减 `main.py` / `xinque.py` 中的职责堆积
- 把 schema 兼容、tool 协议、状态恢复、配置读取这些横切逻辑放到更清楚的边界层
- 清理“为了快速修 bug 加进去的临时逻辑”，保留真正需要的兼容层并标明过渡身份

## 不在本 Sprint 范围

- 不新增新的心理干预能力
- 不新增新的 skill 类别
- 不接外部向量数据库
- 不做完整企业权限系统
- 不做完整 OTel 平台部署
- 不在本 sprint 内解决所有产品页面设计问题

## 设计原则

1. **先统一源头，再删兼容层**  
   不允许“先删适配层，后改源头”，避免再次造成运行中断。

2. **迁移优先于隐式修补**  
   兼容修复可以保留，但 Alembic 必须成为主路径。

3. **环境化优先于散点替换**  
   不只是把一个 `localhost` 改成另一个常量，而是要建立统一读取入口。

4. **收口优先于扩功能**  
   本 sprint 的成功标准不是“功能更多”，而是“结构更稳、边界更清楚、真实运行更少踩坑”。

## 预期产出

- 一个新的迁移链路，覆盖当前主链路依赖的 schema
- 统一后的 Responses-native tool definitions
- 环境化的前后端 API / CORS 配置
- 更清晰的前端页面入口结构
- 精简后的运行时边界
- 对应测试、评估和状态文档更新

## 验收思路

1. 使用旧数据库启动时，不会再因为缺列/缺表而在主链路报错。
2. 新环境可通过 Alembic 正常达到当前 schema，而不是依赖隐式建表。
3. tool 定义源头已经统一，不再依赖运行时协议转换作为正式主路径。
4. 前端/后端不再把开发地址硬编码在业务代码里。
5. 前端页面入口结构比当前更清晰，不再只靠 `#admin` 做功能分叉。
6. `main.py` / `xinque.py` 的横切职责有所下沉，结构更清楚。
7. 自动化测试通过，并补充覆盖本次重构重点。
