# Sprint 21 Spec — 长会话语义压缩与状态治理

## 目标

在完成 Responses API 与 phase-aware 迁移后，进一步升级心雀的长会话治理能力，把当前“最小历史压缩 + Prompt 约束”的方式，提升为更稳定的语义压缩与状态分层治理。

本 Sprint 的目标不是扩业务功能，而是让长期多轮心理支持对话在信息保真、状态清晰和上下文成本之间达到更稳的平衡。

## 背景

当前系统已经有：

- 分层记忆设计理念
- `recall_context` / `search_memory`
- 用户画像与 case formulation
- 最小长会话压缩策略

但目前的长会话治理仍然偏“工程占位”：

- 历史压缩主要依赖简单文本摘要
- 工作记忆、稳定状态、检索记忆之间边界还不够明确
- 长会话持续多轮后，仍可能出现：
  - 重复解释
  - 过度回放历史
  - 阶段漂移
  - 当前目标和历史状态混杂

因此，需要一个独立 Sprint，把长会话治理从“最小可用”升级到“结构化、可持续”。

## 本 Sprint 范围

### 1. 明确长会话状态分层

**做**：

- 把长会话上下文至少拆成以下层：
  - 工作记忆：最近几轮必须保留的原始消息
  - 稳定状态：画像、对齐状态、当前 formulation、最近干预
  - 检索记忆：情景记忆、历史事件
  - 语义压缩摘要：阶段性对话总结

**验收要点**：

- 代码中能看出这些层次，而不是继续把所有内容混进一个 history

### 2. 把简单截断升级为语义压缩

**做**：

- 为长会话建立更稳定的压缩策略：
  - 不只是拼接旧消息
  - 要保留主题、核心困扰、当前阶段、已尝试方法、未完成事项
- 明确摘要何时生成、何时更新、何时作废

**验收要点**：

- 压缩后仍能支撑对话连续性
- 不因压缩而丢掉关键阶段信息或干预上下文

### 3. 加强当前目标与历史状态的分离

**做**：

- 明确哪些信息属于“当前回合必须关注”
- 明确哪些信息只应作为背景参考
- 避免历史高频信息反复压过当前用户新问题

**验收要点**：

- 长会话中更少出现“老问题压住新问题”
- Agent 在多轮后仍能围绕当前目标推进

### 4. 把 `search_memory` / `recall_context` / 压缩摘要协同起来

**做**：

- 明确这三者的协作边界：
  - `recall_context` 负责稳定上下文入口
  - `search_memory` 负责具体历史事件召回
  - 长会话摘要负责持续对话压缩
- 避免三者语义重叠、重复召回和相互打架

**验收要点**：

- 长会话上下文链条更清晰
- 检索与摘要职责不混乱

### 5. 为长会话治理补测试与评估场景

**做**：

- 增加多轮会话压缩测试
- 增加状态分层与恢复测试
- 增加历史问题与当前问题冲突时的回归场景

**验收要点**：

- 长会话治理不是只靠主观体验判断

## 不在范围

- Responses API 协议迁移本身
- 新模型接入
- 新业务模块
- 外部向量数据库替换

## 建议实现策略

1. **先建状态模型**
   - 先定义上下文分层结构
   - 再决定压缩和恢复逻辑

2. **先保连续性，再保压缩率**
   - 心理支持场景里，丢关键上下文比多花一点 token 更危险

3. **优先处理阶段信息**
   - P1-P4 阶段、当前 formulation readiness、最近干预与待跟进事项必须保留

## 验收标准

- 长会话上下文具备明确的分层结构
- 历史压缩升级为更稳定的语义压缩，而不是简单拼接
- 当前目标与历史背景能清晰区分
- `recall_context` / `search_memory` / 摘要三者职责边界清晰
- 至少存在自动化测试或可重复评估覆盖长会话治理

## 影响范围

- `recall_context` / `search_memory` 相关模块
- 长会话相关测试

## 风险与注意事项

- 不要为了压缩而破坏心理支持对话的连续性
- 不要把长会话治理和工具路由混成一个巨大的状态机
- 不要在没有测试的情况下反复调压缩规则
