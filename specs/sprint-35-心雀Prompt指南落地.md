# Sprint 35 - 心雀 Prompt 指南落地

## 背景

项目已经形成了较完整的 Prompt 方法论文档：[`05-心雀Prompt撰写指南-v1.md`](/E:/AI_XinQue_Agent/docs/design/05-心雀Prompt撰写指南-v1.md)。

这份指南不只是写作建议，而是已经定义了心雀在 GPT-5.4 / Responses API 运行时下的关键 Prompt 规则，包括：

- 持久人格与回合控制分层
- `Responses-first` 运行时心智模型
- `instructions / working_contract / working_context` 的职责划分
- 心理支持场景下的“工作性假设”表达
- 默认自然 prose、反条列化
- 短句、低认知负担、探索驱动、半步推进
- 不把普通支持性回复写成教程、文档或建议清单
- 动态注入与本轮覆盖规则
- Prompt 审查清单与反模式

目前这些要求只有一部分已经进入 [`system_prompt.py`](/E:/AI_XinQue_Agent/app/backend/app/agent/system_prompt.py) 和局部测试；仍未形成“指南 -> 运行时 prompt -> evaluator 检查项”的完整闭环。

## 目标

这轮 sprint 的目标，是把 [`05-心雀Prompt撰写指南-v1.md`](/E:/AI_XinQue_Agent/docs/design/05-心雀Prompt撰写指南-v1.md) 中对心雀当前产品阶段有效的要求，系统化落实到：

- 运行时 prompt
- Responses-first contract
- 与用户可见回复相关的评估规则

完成后应达到：

- 心雀 Prompt 不再只依赖零散历史调整，而是明确以这份指南为实现依据
- 文档、代码、测试三层一致
- 心雀用户可见回复更稳定符合“心理咨询师型支持助手”的产品定位

## 范围

### 1. 将 Prompt 指南要求映射到运行时 Prompt 分层

把指南中的核心规则进一步落实到运行时 prompt 分层中，至少覆盖：

- 持久人格
- 输出契约
- 安全契约
- 对话模型契约
- 工具契约
- 完成契约
- 验证契约
- 长会话契约
- 回合级写作控制

要求：

- 不只是“提到这些原则”，而是将其写成对模型具有约束力的 contract
- 明确哪些规则是长期稳定的，哪些是回合级可覆盖的

### 2. 落实 Responses-first Prompt 语义

依据指南中的 Responses-first 规则，继续明确和补足：

- `instructions` 只对当前请求有效
- 跨轮不能把长期契约只押在 `instructions` 上
- `working_contract` 负责跨轮必须可见的最小行为约束
- `working_context` 负责状态、总结、历史与当前焦点
- `previous_response_id` 只负责串接，不替代产品级 contract

要求：

- 心雀 Prompt 体系对这些职责划分保持一致
- 文档和运行时说明不能出现相互冲突

### 3. 落实心理支持场景的用户可见回复风格

将指南中的表层回复规则写实到系统 prompt 中，至少包括：

- 默认自然 prose，而不是列表、编号、小标题块
- 默认短句、低认知负担
- 普通支持性回复优先控制在较轻量的句数与层次
- 默认探索驱动，而不是建议驱动
- 负向 flow 优先按“接住 -> 正常化 -> 缩小问题 -> 一个具体问题”的微节奏推进
- 默认每轮只往前推进半步，不堆叠很多层意思

要求：

- 这部分必须体现心雀作为心理支持产品的特殊约束
- 不得因为 GPT-5.4 偏好结构化，就把普通支持性回复写成文档或清单

### 4. 落实“工作性假设”与非诊断化表达

将指南中的心理理解表达规则进一步写成硬要求，至少包括：

- 所有理解、总结、概念化、归因、机制判断都应写成工作性假设
- 不得把推断写成确定事实
- 不得把支持性总结写成隐性诊断或类型判断
- 在信息不足时应承认不确定性，而不是用确定语气补齐

### 5. 落实 Prompt 审查清单到测试与评估

把指南中的关键审查项，至少落实到以下层级之一：

- prompt contract 测试
- Responses contract 测试
- evaluator 检查项

优先覆盖：

- 是否过早建议化
- 是否默认条列化
- 是否写得过长、过重
- 是否把推断写成事实
- 是否混淆长期 contract 与本轮控制

## 非目标

- 本轮不新增新工具或新 skill
- 不重写业务 tool 的能力边界
- 不要求一次性把所有 evaluator 全部自动化
- 不追求完全复制 Wysa 的产品表达风格

## 验收结果

如果本轮完成，心雀将形成一条明确闭环：

`Prompt 指南 -> 运行时 Prompt -> 测试 / 评估`

也就是说，Prompt 不再只是写作参考，而是成为真正落在产品运行时上的行为规范。
