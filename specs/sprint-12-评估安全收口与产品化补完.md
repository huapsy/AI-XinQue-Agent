# Sprint 12 Spec — 评估、安全收口与产品化补完

## 目标

把心雀从“功能基本齐备的 MVP”推进到“可持续优化的产品候选版本”：建立自动评估基线，补上安全对抗验证，落实隐私与脱敏边界，并完成最小产品化补完。

## 背景

Sprint 01-11 主要解决的是“能不能工作”和“是否具备核心能力”。到了这个阶段，系统通常会面临三个问题：

1. **质量不可量化**：只能靠人工主观判断对话好不好
2. **安全不可回归**：修过的安全问题没有系统性回归样本
3. **产品不可运营**：虽然能跑，但缺少趋势观察、匿名统计或基础管理能力

`product-plan-v2` 的第 13、15 节明确提出了评估、安全和隐私要求，因此本 Sprint 是对 MVP 的一次“质量收口”。

## 本 Sprint 范围

### 1. 自动评估（LLM-as-Judge）

**做**：
- 建立一套最小自动评估流程，定期抽样对话进行打分
- 评估维度建议：
  - 共情质量
  - 安全合规
  - 阶段适当性
  - 干预质量
  - 对齐敏感度
- 评估结果写入评估记录表或结构化文件

**不做**：
- 复杂多模型评委投票
- 在线实时评估

### 2. 安全对抗测试（Red Teaming）

**做**：
- 建立一批固定的安全测试样本，至少覆盖：
  - 危机表达
  - 间接自伤/自杀暗示
  - prompt injection
  - 诱导诊断
  - 诱导药物建议
  - 对齐对抗（激怒、否定、挑衅）
- 形成可重复执行的回归脚本或测试集

### 3. 隐私与脱敏收口

**做**：
- 明确哪些字段必须脱敏后进入 trace / 评估 / 调试输出
- 检查：
  - trace
  - episodic memory
  - summary
  - 干预记录
  - 评估样本
- 增加必要的脱敏辅助函数和测试

**不做**：
- 完整企业级权限系统
- 多租户隔离

### 4. 产品化补完（二选一，优先一个）

可选方案 A（推荐）：
- 用户侧情绪趋势视图
- 基于历史签到和会话数据，展示简洁趋势图

可选方案 B：
- 管理侧匿名统计页
- 查看匿名汇总指标，如会话数、平均轮次、干预完成率、安全层触发率

**推荐理由**：
- 用户侧趋势图更贴近产品感知
- 管理后台可放到后续单独扩展

### 5. 最终基线验收

**做**：
- 基于 Sprint 01-12 的全链路做一次收口验收：
  - 新用户首次对话
  - 回访用户关联历史
  - formulate → match_intervention → load_skill → record_outcome
  - 历史会话查看
  - 危机输入绕过 LLM
  - 普通转介与危机转介
  - alignment 低分兜底
  - episodic memory 检索
  - trace 记录
  - 自动评估可运行

## 用户可感知的变化

如果选择用户趋势图方案，用户将首次看到“我最近一段时间的状态变化”。

如果选择管理统计方案，普通用户无直接变化，但产品团队会首次具备基础运营视角。

无论选择哪个方案，系统整体质量和安全性都会更可控，后续迭代的基础更扎实。

## 不在范围

- 企业级权限系统
- 全量监控平台
- 大规模 A/B 实验平台
- 多模态输入输出
