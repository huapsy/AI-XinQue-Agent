# Sprint 23 Spec — 独立会话状态模型

## 目标

在 `Sprint 22` 已实现 `persisted_session_state` 最小持久化的基础上，进一步把长会话状态从 `TraceRecord.llm_call` 的附属字段升级为独立的会话状态模型。

本 Sprint 的目标是让长会话状态具备更清晰的数据边界、更稳定的恢复入口和更好的后续扩展性，而不是继续扩业务能力。

## 背景

当前系统已经具备：

- `layered context`
- `semantic_summary`
- `current_focus`
- 关键 `stable_state` 快照
- `phase_timeline`
- 从最近一条 trace 恢复 `persisted_session_state`

但当前做法仍然是“最小落地”：

- 长会话状态仍依附在 `TraceRecord.llm_call` 中
- 恢复逻辑依赖“取最近一条 trace”
- 语义摘要没有独立版本边界
- 状态和调试记录混在一起，不利于后续查询、治理和演进

这意味着当前实现适合快速验证，但不适合作为长期状态层的最终形态。

因此，需要一个独立 Sprint，把长会话状态从 trace 附属字段中抽离出来。

## 本 Sprint 范围

### 1. 引入独立的会话状态载体

**做**：

- 为每个会话引入独立状态模型或状态表，至少承载：
  - 最新 `semantic_summary`
  - 最新 `current_focus`
  - 最新关键 `stable_state`
  - 状态更新时间
- 明确“一会话一份当前状态”的读写语义

**验收要点**：

- 长会话状态不再只能从 trace 附属字段恢复
- 状态层与调试层边界清晰

### 2. 将状态恢复逻辑切换到独立状态模型

**做**：

- 优先从独立状态模型恢复当前会话状态
- 保留 trace 附属状态作为兼容回退路径
- 明确恢复优先级：
  - 独立状态模型
  - 兼容回退到最近 trace
  - 最后再做实时拼装

**验收要点**：

- 长会话恢复入口稳定
- 不再默认把 trace 当成状态主来源

### 3. 将状态写入从 trace 附属字段迁移为主写独立状态

**做**：

- 对话完成后优先更新独立状态模型
- trace 中可保留最小副本或引用，用于调试兼容
- 避免同一轮状态写入出现多处不一致

**验收要点**：

- 状态主写路径明确
- trace 不再承担状态主存储职责

### 4. 保持与 `Sprint 21/22` 的 layered context 兼容

**做**：

- 让 `session_context.py` 直接消费独立状态模型
- 保持：
  - `current_focus` 仍以本轮输入更新
  - `semantic_summary` 继续增量合并
  - `stable_state` 不与 profile / formulation / intervention 现有来源冲突

**验收要点**：

- `Sprint 21/22` 的行为保持兼容
- 只是状态源升级，而不是推翻现有运行逻辑

### 5. 补测试与评估场景

**做**：

- 增加独立状态模型读写测试
- 增加恢复优先级测试
- 增加 trace 兼容回退测试
- 增加主流程回归测试

**验收要点**：

- 状态模型升级有自动化验证

## 不在范围

- `phase_timeline` 独立建模
- 状态历史版本浏览 UI
- 新压缩算法
- 新数据库引擎
- 外部缓存或向量数据库

## 建议实现策略

1. **先补模型，再切读路径，再切写路径**
   - 先让独立状态存在
   - 再让恢复逻辑优先读取它
   - 最后切换主写入路径

2. **保留兼容回退**
   - 避免一次性切断 trace 附属状态
   - 便于旧数据平滑读取

3. **状态与 trace 分责**
   - trace 负责调试和协议流
   - state 负责恢复和长期上下文

## 验收标准

- 长会话状态已有独立状态模型承载
- 恢复逻辑优先读取独立状态模型，并保留 trace 回退路径
- 写入逻辑已切到“主写独立状态、trace 仅兼容保留”
- `Sprint 21/22` 的 layered context 继续可用
- 至少存在自动化测试或可重复验证覆盖本次改造

## 影响范围

- `models/tables.py`
- `main.py`
- `session_context.py`
- `xinque.py`
- 相关数据库初始化 / 测试文件

## 风险与注意事项

- 不要让独立状态模型和 trace 状态双写后长期漂移
- 不要在同一 Sprint 同时做状态模型升级和 phase timeline 独立建模
- 要优先保证旧数据可读和恢复路径稳定
